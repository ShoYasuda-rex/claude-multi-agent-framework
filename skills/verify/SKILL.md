## 1. 選択

AskUserQuestion（multiSelect: true）で実行する検証を選ばせる:

> どの検証を実行しますか？

| label | description |
|-------|------------|
| テスト + 静的検証（Recommended） | テスト生成＋実行 + 依存関係・リント・型チェック・ビルド確認 |
| テスト | テスト生成＋実行（ユニット・インテグレーション・E2E） |
| 静的検証 | 依存関係・リント・型チェック・ビルド確認 |
| 動的検証 | ブラウザで画面確認 |

- 「テスト + 静的検証」が選ばれた場合 → テストと静的検証を実行（他の選択は無視）
- それ以外 → 選択されたものだけ実行

---

## 1.5. テスト対象スコープの選択

テストが選択された場合のみ、AskUserQuestion でテスト対象スコープを聞く:

> テスト対象を指定してください

| label | description |
|-------|------------|
| 直近の変更（Recommended） | git diff で検出された変更ファイルのみ対象 |
| ファイル/ディレクトリ指定 | 対象パスを手動で指定 |
| プロジェクト全体 | 全コードを対象にテスト生成 |

- 「直近の変更」→ `git diff --name-only HEAD` を実行し、変更ファイル一覧を `{test_scope}` に設定
- 「ファイル/ディレクトリ指定」→ ユーザーが入力したパスを `{test_scope}` に設定
- 「プロジェクト全体」→ `{test_scope}` は空（スコープ指定なし）

test-checker の prompt に `{test_scope}` が存在する場合、以下を追加する:

```
テスト対象スコープ: 以下のファイルのみを対象にテストを生成・実行してください。
{test_scope}
```

## 2. コンテキスト収集

サブエージェントに渡すプロジェクト文脈を収集する。

以下のファイルを Read で読む（存在しないものはスキップ）:

1. `docs/CORE.md` — 誰の課題を解決するプロダクトか
2. `docs/ARCHITECTURE.md` — データモデル・API設計・認証方式・外部連携

読み取った内容から、以下の `{project_context}` ブロックを組み立てる:

```
## プロジェクト文脈
- プロダクト概要: （CORE.md の「一言で言うと」）
- 技術スタック: （ARCHITECTURE.md の技術スタックテーブル）
- データモデル: （ARCHITECTURE.md の主要モデル一覧）
- 認証方式: （ARCHITECTURE.md の認証・権限設計）
- 外部連携: （ARCHITECTURE.md の外部API・サービス）
```

CORE.md も ARCHITECTURE.md も存在しない場合、`{project_context}` は空文字にする。

## 3. 実行

選択された検証をサブエージェントとして起動する。

| 検証 | subagent_type |
|------|--------------|
| 静的検証 | code-checker |
| テスト | test-checker |
| 動的検証 | visual-checker |

### 実行順序

**静的検証とテストは直列実行**する（静的検証 → テスト）。動的検証は並列で同時起動してよい。

1. **まず静的検証（code-checker）を起動**する（`run_in_background: true`）
   - 動的検証も選択されている場合、同じメッセージで visual-checker も並列起動する
2. **静的検証の完了を待つ**
3. 静的検証で **ビルドエラーや型エラーがある場合 → テストを実行せず報告**（壊れたコードのテスト生成は無意味なため）
4. 静的検証がパスした場合 → **test-checker を起動**する（`run_in_background: true`）

### 理由

テスト生成は現在の実装コードを正としてテストを作るため、バグがあるとバグった動作を正としたテストになる。静的検証で先に問題を検出・報告し、修正後にテストを生成することで、正しいコードに対するテストが生成される。

### prompt

各サブエージェントの prompt に `{project_context}` を先頭に付与する:

```
{project_context}

上記のプロジェクト文脈を踏まえて検証してください。
```

## 4. 結果の統合

全サブエージェントの完了を TaskOutput で待ち、結果を統合する。

### 統合ルール

複数の検証結果を以下の優先順位で統合し、「まずこれを直せ」の形でユーザーに提示する:

- 優先順位: **ビルドエラー > テスト失敗 > リント警告 > UI崩れ**
- ビルドエラーがある場合、visual-checkerの結果を「参考」に格下げ（ビルドが通らない状態でのUI検証は信頼度が低い）
- 同一ファイルに対する複数の指摘は集約して提示する

### 動的検証のスクショ確認（visual-checker実行時のみ）

visual-checker の結果に含まれるスクリーンショットのパスを全て Read で読み込み、自分の目で視覚分析する:
- レイアウト崩れ、要素の欠落、配置ずれ
- テキストの切れ、重なり、はみ出し
- レスポンシブ対応の問題（各viewport間の比較）

検出した問題は最終報告の動的検証セクションに追記する。

### 判定

- 全パス → ステップ6へ
- テスト失敗あり → ステップ5へ
- 静的検証・動的検証のみ失敗（テストは全パス） → ステップ6へ

## 5. 自動修正ループ（テスト失敗時のみ）

テスト失敗がある場合、**ユーザーに確認せず自動で修正→再実行を繰り返す**。

### 修正対象の判別

失敗内容を分析し、以下に分類する:

| 分類 | 例 | 対応 |
|------|---|------|
| spec側の不備 | セレクタ不一致、ラベル不一致、テストデータ不足、typo、Ambiguousエラー | specファイルを修正 |
| 本番コードのバグ | ロジックエラー、未実装機能、例外発生 | 修正せず報告のみ |

### ループ手順

1. 失敗したテストの原因を分析する
2. **spec側の不備** → 対象のspecファイルと実装コード（ビュー・モデル・スキーマ）を読み、実装に合わせてspecを修正する
3. **本番コードのバグ** → `{production_bugs}` に記録し、修正しない
4. 修正したspecのみを対象に test-checker を再実行する
5. まだ失敗があれば 1 に戻る

### ループ上限

- **最大3回** まで繰り返す
- 3回で解消しない失敗は `{unresolved_failures}` に記録してステップ6へ進む

### spec修正時の必須チェック

修正前に必ず以下を確認すること（推測で書かない）:

- ビュー/テンプレートの **実際のラベル文言**
- モデル/マイグレーションの **実際のカラム名**
- フォームの **セレクタの一意性**（同名ラベルが複数ないか）
- フィクスチャ/テストデータが **ビューの参照する全フィールド** を含んでいるか

## 6. 最終報告

## 報告フォーマット

```
## /verify 完了

### テスト（test-checker）
[結果サマリー]
- 自動修正: X件（ループN回で全パス）
- 本番バグ: X件（要対応）
- 未解消: X件（3回で解消せず）

### 静的検証（code-checker）
[結果サマリー]

### 動的検証（visual-checker）
[結果サマリー]

### 総合判定
- CRITICAL: X件
- WARNING: X件
- INFO: X件
```

※選択されなかったツールのセクションは省略する
※自動修正・本番バグ・未解消が0件の行は省略する

## ルール

- 最低1つは選択されていること
- **静的検証 → テストは直列実行**（静的検証パス後にテスト生成）。動的検証のみ並列可
- 静的検証でビルドエラー・型エラーがある場合、テスト生成はスキップする
- 結果の統合・報告のみメインが行う（検証自体はサブエージェントに任せる）
- テスト失敗時はユーザーに聞かず自動修正する（本番コードのバグは除く）
- spec修正時は必ず実装コードを読んでから修正する（推測で書かない）

# /research

ユーザーが指定したテーマを、複数の視点からサブエージェントで並列調査し、統合レポートとして報告する。

## フロー

1. ユーザーに調査テーマを聞く（1回だけ）
2. テーマを分析し、そのテーマに最適な調査視点を**最低5つ**自分で設計する（テーマの複雑さに応じて増やしてよい）
3. サブエージェントをTaskで**並列起動**
4. 全員の結果を統合し、レポートファイルを作成して終了

## レポートファイル

- 保存先: プロジェクトのルートディレクトリ
- ファイル名: `{YYYY-MM-DD}_{テーマを英語kebab-case}-report.md`
  - 例: `2026-03-06_obsidian-vs-remory-report.md`
- 内容は下記「統合レポートフォーマット」に従う
- ファイル作成後、チャットにもレポートの要約を報告する

## 視点の設計

視点は固定ではない。テーマごとに「このテーマを理解するために必要な切り口」を最低5つ考えて決める。テーマが複雑・多面的なら7〜8視点でもよい。

**設計の指針:**
- 視点同士が互いに重複しないこと（MECE寄り）
- 少なくとも1つは「主流と逆の見方」を含めること（盲点防止）
- 各視点に明確な調査対象と観点を設定すること
- 視点名はテーマに合わせた自然な名前にすること（「視点A」のような汎用名は禁止）

**例（あくまで例。テーマに応じて自由に変えること）:**

テーマ「リモートワークの生産性」の場合:
1. 定量データ — 生産性の測定結果・統計調査
2. 従業員の実態 — 当事者の声・実際の働き方
3. 企業の取り組み — 制度設計・マネジメント手法
4. ツール/環境 — 技術的な解決策・ワークスペース設計
5. 反リモート論 — オフィス回帰の動き・リモートの限界

テーマ「次に学ぶプログラミング言語」の場合:
1. 言語比較 — 主要候補の特徴・エコシステム
2. 求人/市場 — 需要・年収・採用トレンド
3. 学習体験 — 学習コスト・教材・コミュニティ
4. 実用事例 — 各言語で作られた有名プロダクト
5. 衰退リスク — 廃れつつある言語・過大評価されている言語

## ルール

1. **聞くのは最初の1回だけ。** テーマ確認後は一切質問しない。
2. **全視点を必ず並列起動。** 順番に実行しない。
3. **統合せよ。** 全視点の結果を突き合わせ、重複排除・重要度順に整理して報告。
4. **WebSearchを積極的に使え。** 各調査員はWebSearchで実データを集めること。推測で埋めない。

## 各調査員への指示

以下のテンプレートに、設計した視点の情報を埋めて各サブエージェントに渡す。

```
あなたは「{視点名}」の視点から以下のテーマを調査する調査員です。
WebSearchを使って実際のデータ・事例・情報を集めてください。
推測や一般論だけで埋めず、具体的なソース・事例を示すこと。

調査テーマ: {theme}
あなたの視点: {視点名}
調査の観点:
{そのテーマ・視点に合わせた具体的な観点を3-5個}

報告フォーマット:
- 発見した事実（出典付き）を箇条書きで列挙
- 最後に「この視点からの要約」を3行以内で
```

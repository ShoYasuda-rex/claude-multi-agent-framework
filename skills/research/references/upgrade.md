
# /upgrade-team

5つの視点からサブエージェントで並列調査し、結果を統合してレポートを出力する。

## 調査員（サブエージェント — 使い捨て）

| 調査員 | 視点 | 読むもの | 見ないもの | subagent_type |
|--------|------|---------|-----------|---------------|
| **A** | 現物体験（Playwrightで実操作） | 現物 | docs | general-purpose |
| **B** | ユーザー理解（文脈・ワークフロー・生の声） | WebSearch | コード | general-purpose |
| **C** | 競合・市場（類似サービス・トレンド） | WebSearch | コード | general-purpose |
| **D** | 再構想（docs全部読んで白紙から設計） | BRAINSTORM + CORE + ARCHITECTURE | コード・現物 | general-purpose |
| **E** | ビジュアル品質評価（調査→Aのスクショで現物判定） | WebSearch → Aのスクショ | コード | general-purpose |

## ルール

1. **体感インパクトで優先度をつけろ。** 判定基準: 改善したらユーザーが違いに気づくか？
2. **統合せよ。** 5視点の結果を突き合わせ、重複排除・優先度付けして報告。
3. **聞くのは起動時だけ。** テーマ確認のみ。それ以降は聞かない。

## フロー

1. ユーザーに調査テーマを確認（指定なしならプロダクト全体）
2. docs/CORE.md があれば読む
3. A/B/C/D/E のサブエージェント（subagent_type: general-purpose）を `run_in_background: true` で **1つのメッセージ内で同時に** 起動する
   - Eは前半（WebSearch調査）を先に進め、A完了後にスクショを見て現物評価する
   - Aは調査中にスクリーンショットを `check_log/screenshots/` に保存すること
4. 全サブエージェントの完了を待ち、結果を統合してレポートファイルを作成

## レポートファイル

- 保存先: `../reports/{プロジェクト名}/` ディレクトリ（プロジェクト名 = カレントディレクトリのフォルダ名。なければ作成）
- ファイル名: `{YYYYMMDD}_{HHmm}_upgrade.md`
  - 例: `20260309_1430_upgrade.md`
- 内容は下記「統合報告フォーマット」に従う
- ファイル作成後、チャットにもレポートの要約を報告する

## 調査員への指示

### A（UX調査）
```
このプロダクトの主要画面をPlaywrightで操作し、ユーザー体験を調査してください。
調査コードは書かないでください。Playwright MCPツール（browser_navigate, browser_snapshot, browser_click等）を直接使ってください。

観点:
- ユーザーフローで引っかかる箇所
- 「あったら嬉しい」のに存在しない機能・インタラクション
- モバイル/デスクトップでのレイアウト崩れ（体験に影響するレベルのみ）
- インタラクションの違和感（ローディング、フィードバック不足等）
- エラー時の体験

重要: 「改善したらユーザーが体感できるもの」を優先して報告せよ。余白1pxのズレや色コードの微差など開発者しか気づかないレベルの指摘は不要。
報告フォーマット: 発見（何が起きているか）→ 体感への影響（ユーザーはどう困るか/何が嬉しくなるか）→ 改善イメージ

スクリーンショットはPCビューポート（デスクトップ幅）のみで撮影すること。モバイル・タブレットのリサイズは不要。
スクリーンショットは必ず `check_log/screenshots/` に保存すること（調査員Eが後から参照する）。

対象テーマ: {theme}
ローカルサーバー: {url}
```

### B（ユーザー理解調査）
```
WebSearchでこのプロダクトのターゲットユーザーを調査してください。コードは読まないでください。

観点:
- ターゲットユーザーの日常のワークフロー・行動パターン
- このプロダクトの前後で何をしているか（前後の文脈）
- SNS・レビュー・フォーラムでの生の声（不満・要望・工夫）
- ユーザーが自力で解決している課題（既存ツールの組み合わせ、手作業等）

重要: 「この機能があればユーザーの行動が変わる」レベルの発見を優先せよ。プロダクトの内部品質（コード・設計）の話は不要。
報告フォーマット: 発見（ユーザーはどう行動しているか）→ 課題（何に困っている/何が面倒か）→ 機能アイデア（こういう機能があれば解決する）

対象テーマ: {theme}
プロダクト概要: {product_summary}
```

### C（競合・トレンド調査）
```
WebSearchで類似サービス・業界動向を調査してください。

観点:
- 類似サービス・競合の特徴的なUX/機能
- この分野の最新トレンド・ベストプラクティス
- ユーザーが期待しそうな機能・体験

重要: 「取り入れたらユーザーが体感できる変化」を優先して報告せよ。技術的なアーキテクチャ比較やコード構成の話は不要。
報告フォーマット: 発見（競合では何がある/トレンドは何か）→ 体感への影響（取り入れたらユーザーは何が嬉しいか）→ 導入イメージ

対象テーマ: {theme}
プロダクト概要: {product_summary}
```

### E（ビジュアル品質評価）

2段階で進める。コードは読まないこと。

**STEP 1（Aと並列で即開始）: WebSearchで品質水準を調査**
```
WebSearchでこのジャンルのプロダクトのビジュアル品質を調査してください。

観点:
- 同ジャンルで評判の良いサービスが使っているビジュアル要素（3D、イラスト、アイコン、アニメーション等）
- それらのアセットの品質水準（どのレベルなら「プロっぽい」と感じるか）
- ビジュアル演出のトレンド（マイクロアニメーション、スクロール演出、パーティクル等）

対象テーマ: {theme}
プロダクト概要: {product_summary}
```

**STEP 2（A完了後）: Aのスクショで現物を評価**
```
check_log/screenshots/ にあるスクリーンショットをReadツールで読み、STEP 1の調査結果と比較して現物のビジュアル品質を評価してください。

評価観点:
- 3Dモデル/イラスト/アイコン等のアセット品質（しょぼくないか、統一感があるか）
- アニメーション・演出の有無と品質
- 競合と比べて見劣りする箇所

報告は以下の2カテゴリに分けること:
- 🔧 **コードで改善可能**: アニメーション追加、アイコン差し替え、CSS演出、レイアウト調整等
- 🎨 **アセット差し替えが必要**: 3Dモデル、イラスト、写真素材等の品質問題 → ユーザーが素材を用意する必要がある

報告フォーマット: 現状（何がどう見えるか）→ 問題（競合水準と比べて何が劣るか）→ 改善案（カテゴリ明記）
```

### D（再構想）
```
以下のドキュメントを全て読んでください。ただしコードや現物は一切見ないでください。

読むもの:
- docs/BRAINSTORM.md（最初の発想）
- docs/CORE.md（誰の何の課題を解くか）
- docs/ARCHITECTURE.md（技術スタック・構成）

あなたはこのプロダクトを今からゼロベースで設計する立場です。
既存の実装は存在しないものとして、「この課題を、この技術スタックで、白紙から作るならどう作るか？」を考えてください。

観点:
- BRAINSTORMにあったが実現されていないアイデア（忘れられた着想）
- CORE.mdの課題に対して、もっと直接的な解き方はないか
- ARCHITECTURE.mdの技術スタックで実現可能だが、誰も思いつかなそうな機能
- 「この課題を抱えるユーザーが本当に欲しいもの」を原点から考える

重要: 既存の実装の改善ではなく、「あるべき姿」から逆算して提案せよ。実装の難易度は考慮しなくてよい。
報告フォーマット: 提案（何を作るか）→ なぜ（どの課題をどう解くか）→ ユーザー体験（使うとどうなるか）

対象テーマ: {theme}
```

## 統合報告フォーマット

```
## プロダクト改善レポート
### テーマ: {theme}

### 発見事項（体感インパクト順）
1. **[High]** 〇〇 — 改善したらユーザーが即座に気づくレベル（根拠: A/B共通）
2. **[Medium]** 〇〇 — 体験が明確に良くなる（根拠: C/Dから示唆）
3. **[Low]** 〇〇 — あれば嬉しいが優先度は低い

### 改善提案
| # | 提案 | 体感の変化 | 根拠（調査員） |
|---|------|-----------|---------------|
| 1 | ... | ユーザーは○○が変わったと感じる | A, B |
| 2 | ... | ユーザーは○○できるようになる | C |
```

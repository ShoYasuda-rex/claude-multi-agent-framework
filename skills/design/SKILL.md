# Design

5つ以上のデザインパターンを生成して比較し、確定したらそのまま全画面を実装する。
初期設計でも開発途中でも使える。

---

## 起動時の動作

### Step 0: コンテキスト収集

以下を読み込む（存在するもののみ。なくてもOK）:

1. `docs/CORE.md`
2. `docs/ARCHITECTURE.md`
3. `CLAUDE.md`
4. 既存のCSS / デザイントークン（プロジェクトにあれば）

---

## Phase 1: ヒアリング

### Step 1: 対象の特定

> どの画面（or コンポーネント）のデザインを探る？
>
> 例: トップページ / ダッシュボード / 設定画面 / カード一覧 / ナビゲーション など

ユーザーが引数で指定済みなら聞かない。

### Step 2: 制約・好みの確認（軽く）

> 何か制約や好みはある？（なければスキップ）
>
> 例: ダーク系がいい / モバイル重視 / 情報量多め / 既存デザインに合わせたい

「なし」「特になし」→ 自由に多角的に出す。

---

## Phase 2: パターン生成

### Step 3: 設計方針の決定

対象画面に対して、**異なるデザイン哲学**から5つ以上のアプローチを設計する。

**アプローチの軸（例）** — 対象に応じて最適な軸を選ぶ:

- レイアウト: シングルカラム / マルチカラム / グリッド / フリーフォーム
- 密度: ミニマル（余白多め） / 高密度（情報一覧性重視）
- ナビゲーション: サイドバー / タブ / ボトムナビ / ハンバーガー
- トーン: ビジネス / カジュアル / テック / ナチュラル / ブルータル
- インタラクション: カード型 / リスト型 / タイムライン型 / 会話型
- カラー: ライト / ダーク / アクセントカラー違い / モノクロ

**ルール**:
- 最低5パターン。対象が複雑なら7〜8出す
- 各パターンは**明確に違う方向性**にする（色違いだけはNG）
- 各パターンに設計意図を1〜2行で書く

### Step 4: 静的ページ生成

各パターンを HTML + CSS で作成する。

**方針**:
- 見た目だけ。JS は最小限（タブ切り替え等の表示制御のみ許可）
- ダミーデータで埋める（空の画面にしない）
- レスポンシブ対応（モバイルファースト）
- 各パターンは独立した HTML ファイル
- **デザイントークン（色・フォント・余白・角丸など）をCSS変数で定義する**

**出力先**: `design/`

```
design/
├── index.html          ← 一覧ページ（全パターンへのリンク + 設計意図）
├── pattern-1.html      ← パターン1
├── pattern-2.html      ← パターン2
├── pattern-3.html      ← パターン3
├── pattern-4.html      ← パターン4
├── pattern-5.html      ← パターン5
├── pattern-6.html      ← （必要に応じて追加）
└── styles/
    ├── common.css      ← 共通リセット・ベーススタイル
    ├── pattern-1.css   ← CSS変数でデザイントークン定義
    ├── pattern-2.css
    └── ...
```

**index.html の構成**:
- タイトル: 「Design: {対象画面名}」
- 各パターンのサムネイル的な説明:
  - パターン名（例: 「ミニマル・余白重視」）
  - 設計意図（1〜2行）
  - リンク（別タブで開く）

### Step 5: ブラウザで確認

1. `browser_navigate` で `design/index.html` を開く
2. `browser_snapshot` で一覧ページを確認
3. 表示に問題があれば自動修正

> 5パターン作った。一覧ページを開いたので、気になるものを見て。
> どれが近い？ 混ぜたい要素があれば言って。

---

## Phase 3: 絞り込み・調整

ユーザーのフィードバックに応じて対応する。

| フィードバック | 対応 |
|--------------|------|
| 「3番が近い」 | その方向性で微調整パターンを2〜3追加 |
| 「2と5を混ぜたい」 | 要素を組み合わせた新パターンを生成 |
| 「全部違う」 | 何が違うかヒアリング → 新しい軸で再生成 |
| 「ここを変えたい」 | 選んだパターンを修正してブラウザで再確認 |
| 「これで決まり」 | Phase 4 へ |

**追加パターンも `design/` に追加する**（既存は消さない）。
index.html も更新する。

**軌道修正**:

| 逸れた内容 | 戻し方 |
|-----------|--------|
| ロジック・機能の話 | 「機能は実装フェーズで。今はデザインの方向性を固めよう」 |

---

## Phase 4: 実装

デザインが確定したら、そのデザインをベースにプロダクト全体を実装する。

### Step 1: 実装方針の確認

> デザイン確定: {選んだパターン名}。これをベースに実装に入る。

ARCHITECTURE.md の技術スタックに基づいて、実装の進め方を簡潔に提示する:

- 何を作るか（画面一覧）
- どの順で作るか

> この順で進めていい？

### Step 2: 実装

確定したデザイン（CSS変数・レイアウト・配色・タイポグラフィ）を踏襲しながら、全画面を実装する。

**ルール**:
- 確定パターンの CSS 変数（デザイントークン）をプロダクトのCSSに移植する
- 各画面は確定パターンと同じデザインテイストで統一する
- docs/CORE.md、docs/ARCHITECTURE.md に従って実装する
- CLAUDE.md の実装タスクのフローに従う（依存関係確認 → 設計 → 実装）
- **確定したデザインを勝手に変えない**

### Step 3: 完了報告

> 実装完了。`/verify` で品質チェックしてね。

---

## ルール

1. **最低5パターン** — 少ないと比較にならない
2. **明確に違う方向性** — 色違い・フォント違いだけのバリエーションは出さない
3. **デザイントークンをCSS変数で定義** — 実装時に全画面に展開するため
4. **一覧ページ必須** — 比較しやすさが命
5. **既存デザインがあれば尊重** — 途中参加の場合、既存のデザイントークンを読んで統一感のあるパターンも含める
6. **CORE.md なしでも動く** — 「この画面のデザインを探りたい」だけで起動できる
7. **確定したデザインを崩さない** — 実装フェーズでデザインを勝手に変えない
8. **design/ は実装後も残す** — デザインの原点として参照できるように

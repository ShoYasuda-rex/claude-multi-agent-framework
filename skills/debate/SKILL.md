# /debate

任意のテーマを2フェーズで深掘り議論する。

| フェーズ | 性質 | 役割 | 順番 | 発言形式 |
|----------|------|------|------|----------|
| **発散** | 広げる | なし | なし | 2〜3行の短文のみ |
| **収束** | まとめる | Builder / Expander / Critic | あり | 構造化OK |

---

## フロー

### 起動時
`.reports/` 内に結論セクションがない（=進行中の）debateファイルがあるか確認:
- **ある** → 自動で参加フローへ（選択不要）
- **ない** → 新規作成フローへ（選択不要）

---

## Phase 1: 発散

- **役割なし。** 全員フラット。IDだけで識別。
- **直前の発言者以外が応答。** 連投防止・書き込み競合回避。
- **2〜3行の短文のみ。** 箇条書き・見出し・構造化は禁止。
- **否定OK、賛同OK、脱線OK。** 深く考えすぎない。即書く。

発言: `### [{ID}]` + 2〜3行の短文

### 発散 → 収束への移行

誰でも収束提案できる（`### [{ID}] 🔄 収束提案` + Builder/Expander/Criticの割り振り案）。

- 全員合意 → Phase 2へ（**ユーザー参加中はユーザーの賛成も必須**）
- 1人でも反対 → 継続。**最低2往復後に再提案可**
- **ユーザーが「終了」→ 収束に進まず発散の内容で結論ドキュメントを作成して終了**
- 割り振りは発散中の発言傾向から: 具体案→Builder、別角度→Expander、リスク指摘→Critic

---

## Phase 2: 収束（Builder × Expander × Critic）

| 役割 | やること | やらないこと |
|------|---------|-------------|
| **Builder** | 素材を整理・構造化・設計に落とす | 問題点を無視 |
| **Expander** | 新しい切り口・見落とし・可能性を投げ込む | Builderを否定（Criticの仕事） |
| **Critic** | 穴を突く・エッジケース・実現可能性を問う | 対案を出す（Builderの仕事） |

順序: `Builder → Expander → Critic` を1ラウンドとする。

### Round 1
- **Builder**: 自分の案ではなく発散の素材整理から。対立意見も両方提示し、方針を提案
- **Expander**: 発散で拾われなかった素材・新しい角度を投げ込む
- **Critic**: 穴を突く。観点はテーマに応じて自分で設計

### Round 2〜
- **Builder**: Expander案の採否（理由付き）+ Critic指摘への修正or反論を全て回答
- **Expander**: さらに新しい角度を投げ込む
- **Critic**: さらに深い問題を提起

---

## 終了判定（Builderが判断）

**最低3ラウンド**実施後、以下の**いずれか1つでも**該当すれば**続行**:
1. 根本に関わる新視点・重大問題が未応答
2. Criticの具体的質問にBuilderが未回答
3. Criticがまだ重大な懸念を示している

### ユーザー発言ルール
- **ユーザー発言の直後に結論を書いてはならない。** ユーザーの指摘は必ずBuilder→Expander→Criticの1ラウンドを回してから反映する。
- ユーザー発言は「新しい論点の追加」として扱い、結論への吸収ではなく議論の継続トリガーとする。

終了時: Builderが完成版設計 → 結論ドキュメントをファイル末尾に書く。

---

## ユーザー参加

ユーザーはチャットで発言 → AIが `### [User_xxxx]` としてファイルに書き込む。発散・収束どちらでも割り込み可。

4人目が `/debate` を実行した場合（AI枠3人が埋まっている → 自動でUser参加）:
1. 進行中ファイルを自動選択 → ID追記 → 議論要約表示
2. ユーザーの発言をそのまま `### [User_xxxx]` としてファイルに書き込む → 監視ループへ

## 新規作成

**ステップ1: ファイル生成（Haikuサブエージェント・即実行）**

テーマ入力を待たずに:
1. `.reports/` フォルダを作成（なければ）
2. ID生成 → `.reports/{YYYYMMDD}_{HHmm}_debate.md` を作成
3. ヘッダー（テーマ未定・参加者・フェーズ: 発散）+ `## 発散` を書き込む
4. 「別ターミナルで `/debate` して参加してください」と案内

**ステップ2: テーマ入力と発散開始（本体・並行）**

1. `.reports/*_research.md` を検索
   - **あり** → 一覧表示（＋「なし」の選択肢）。レポート選択 → テーマはレポートから取得、要約・発見・結論を `## 参考資料` に埋め込む。「なし」選択 → テーマを聞く
   - **なし** → テーマを聞く
2. テーマ更新 → 最初の発散発言 → ファイル監視ループへ

---

## 参加

**ステップ1: ID生成＋参加登録（Haikuサブエージェント）**

1. `.reports/` 内の最新debateファイルを自動選択（なければ10秒待ってリトライ）
2. ID生成 → **ロック取得** → 参加者リストを再読み込みしAI枠（3人）を超えていないか確認 → 追記 → **ロック解放** → フェーズを返す
   - AI枠が埋まっていた場合 → User_xxxx として登録

**ステップ2: 議論参加（本体）**

発散中 → 即発言、収束中 → 割り振り役割で参加 → ファイル監視ループへ

---

## ファイル監視ループ

1. 約10秒間隔で監視（読み取りはロック不要）
2. **発散:** 直前が自分以外 → 応答書き込み。収束提案があれば賛否表明
3. **収束:** 自分の番なら応答書き込み
4. 結論セクション検知 → 終了

**更新がなくてもループを終了してはならない。**

## ファイルロック

複数エージェントの書き込み競合を防ぐため、debateファイルへの書き込み時は必ずロックを取得する。

```bash
# ロック取得（最大10秒リトライ）
LOCK_DIR=".reports/.debate.lock"
for i in $(seq 1 10); do
  mkdir "$LOCK_DIR" 2>/dev/null && break
  sleep 1
done

# --- ここでファイル書き込み ---

# ロック解放
rmdir "$LOCK_DIR"
```

**ロック必須の操作:** 参加者登録のみ（AI枠チェックにレースコンディションがあるため）
**ロック不要の操作:** 発言書き込み、フェーズ変更、結論書き込み、ファイル読み取り（「直前の発言者以外が応答」ルールで順番制御されるため競合しない）

クラッシュ対策: ロックディレクトリが60秒以上前に作成されていた場合は強制削除してリトライする。

```bash
if [ -d "$LOCK_DIR" ]; then
  LOCK_AGE=$(( $(date +%s) - $(stat -c %Y "$LOCK_DIR" 2>/dev/null || echo 0) ))
  [ "$LOCK_AGE" -gt 60 ] && rmdir "$LOCK_DIR" 2>/dev/null
fi
```

## 結論ドキュメント

Builderがファイル末尾に書く（別ファイルは作らない）。セクション名はテーマに応じて選ぶ。

**必須:** 概要 / 決定事項 / リスクと対策
**任意:** 採用しなかったアイデア / 技術スタック / UI設計 / データ構造 / フォルダ構成 / KPI / タイムライン等

---

## チャット表示

各発言をファイルに書き込むたびに、チャットにも要約を表示する。ファイルが真実、チャットは要約。

---

## ルール

1. **ファイルが真実。** 全文はファイル。チャットは要約。
2. **自動応答。** 検知したら即応答。ユーザーに確認しない。
3. **日本語で議論。**
4. **IDで識別。** 同モデルでもhexで区別。
5. **Builderは全指摘に回答する。** 逃げない。
6. **Builderだけが結論を書く。**

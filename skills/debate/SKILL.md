# /debate

任意のテーマを2フェーズで深掘り議論する。

| フェーズ | 性質 | 動作モード | 発言形式 |
|----------|------|-----------|----------|
| **発散** | 広げる | 3体+検証者 常駐・並列・自由発言 | 2〜3行の短文のみ |
| **収束** | まとめる | 3体+検証者 常駐・順番制 | 構造化OK |

---

## アーキテクチャ

**ホスト + 参加者構成。フェーズで動作モードが変わる。**

| チャット | 役割 | 動作 |
|----------|------|------|
| **ホスト** | 進行管理 | 4体をバックグラウンド起動 → 監視ループで `## 収束` `## 結論へ` を書く → 結論表示 |
| **参加者** | ユーザー/AIが発言 | `/debate-join` で入室。発言をファイルに書き込む。閲覧はファイルを直接開く |

```
ホスト: テーマ入力 → ファイル生成 → 4体起動 → 監視ループ（発散→収束を通す）→ 結論表示
参加者: /debate-join → 入室 → 発言
```

---

## フロー

このスキルはホスト専用。参加は `/debate-join` を使う。

---

## ホストフロー

### ステップ1: テーマ入力

1. ユーザーにテーマを聞く
   - ユーザーが「レポートから」等と指示した場合のみ → `../reports/{プロジェクト名}/*_research.md` を検索・一覧表示。選択されたレポートの全文を `## 参考資料` に埋め込む（要約しない）
   - ユーザーがテーマと共に参考情報を提供した場合も、原文をそのまま `## 参考資料` に記載する（要約しない）

### ステップ1.5: 観点設計

1. テーマを分析し、6つの多角的観点を生成する（「観点設計」セクション参照）
2. 3体に2観点ずつ割り当てる

### ステップ2: ファイル生成

1. **レポート保存先**: `../reports/{プロジェクト名}/`（プロジェクト名 = カレントディレクトリのフォルダ名。`basename "$(pwd)"` で取得）
2. 上記ディレクトリを作成（なければ）
3. `../reports/{プロジェクト名}/{YYYYMMDD}_{HHmm}_{theme}_debate.md` を作成（themeはテーマを短縮した英語スラッグ）。ヘッダー（テーマ: 確定済み・参加者: なし・フェーズ: 発散）+ 参考資料（あれば）+ `## 発散` を書き込む
4. AI 4体（`Claude-A` / `Claude-B` / `Claude-C` / `Claude-V`）を参加者リストに追記
5. 「参加する場合は別チャットで `/debate-join user` を、AI発言を追加する場合は `/debate-join` をどうぞ」と案内し、AI 4体の議論を即開始

### ステップ3: 4体起動 + ホスト監視（発散→収束を通す）

4体のエージェントを **`run_in_background: true` で** 並列起動する（Agent tool × 4）。
ホストは即座に制御を取り戻し、自分で監視ループに入る。
4体は `## 結論` まで解散しない。発散→収束をまたいで常駐する。

各エージェント（Claude-A/B/C）に渡す情報:
- 自分のID
- debateファイルのパス
- **発散フェーズの担当観点（ステップ1.5で設計した2観点）**
- **⚡ そのエージェント固有の前提情報（情報の非対称性）** — 他のエージェントには渡さない
- 発散フェーズのルール（短文のみ、反論義務、同意禁止）
- 収束フェーズのルール（役割・順番制）
- **「即座に議論を開始。`## 収束` 検知で収束モードに切り替え。`## 結論` 検知で終了」** という指示

検証者（Claude-V）に渡す情報:
- 自分のID（`Claude-V`）
- debateファイルのパス
- **検証者のルール（下記「検証者」セクション参照）**
- WebSearch / WebFetch ツールの使用許可
- **「議論には参加しない。事実検証のみ行う。`## 結論` 検知で終了」** という指示

```
議論エージェント（Claude-A/B/C）の動作:
  ループ {
    ファイルを読む
    `## 結論` 検知 → 終了

    `## 収束` がない → 発散モード:
      直前の発言者が自分以外 →
        直前の発言を読み、それに対する反論・疑問・条件付き補足を短文1発言で追記
        （同意から始めない。必ず摩擦を含める）

    `## 収束` がある → 収束モード:
      自分の役割を確認（`## 収束` 直後の割り当て行から）
      直前の発言者の役割を確認
      順番判定（批評者 or User の後→設計者、設計者の後→探索者、探索者の後→批評者）
      自分の番 → 1発言を追記
      設計者のみ: `## 結論へ` 検知 → `## 結論` ドキュメントを書く

    sleep
  }

検証者（Claude-V）の動作:
  ループ {
    ファイルを読む
    `## 結論` 検知 → 終了

    新しい発言を確認
    事実に基づく主張・前提を検出
    → WebSearch / WebFetch で裏取り
    → 誤りや不正確な点があれば `### [Claude-V]` で指摘を追記
    → 正確であれば何も書かない（沈黙）

    sleep
  }
```

**ホストの監視ループ（重要）:**

ホストは4体を起動した直後に自分で監視ループに入る。
4体のエージェントが全て終了するまで以下を繰り返す。

```
ループ {
  sleep 10秒
  debateファイル末尾を読む
  エージェントの output_file を確認（全員終了したか）

  発散中で移行基準を満たした（「発散→収束への移行」参照）
  → ファイルに `## 収束` + 役割割り当て行 + `### 📌 Round 1 焦点: 構造整理` を書き込む → ループ継続
  収束中で新しいラウンドが始まる
  → ファイルに `### 📌 Round N 焦点: {テーマ}` を書き込む（Round 2は「実現可能性・リスク」、Round 3〜はホストが状況を見て設定）→ ループ継続
  結論移行基準を全て満たした（「終了判定」参照）
  → ファイルに `## 結論へ` を書き込む → ループ継続（設計者が結論を書くのを待つ）

  全エージェント終了を確認 → break
}
```

**役割割り当ての書き方:**

ホストが `## 収束` を書く際、直後に1行で役割を割り当てる。発散中の発言傾向から判断する（具体案→設計者、別角度→探索者、リスク指摘→批評者）。検証者は固定（Claude-V）。

```
## 収束
Claude-A=設計者, Claude-B=探索者, Claude-C=批評者, Claude-V=検証者
```

### ステップ4: 結論表示

結論セクションを読んでチャットに表示して終了。

## Phase 1: 発散

- **役割なし。** 全員フラット。IDだけで識別（検証者除く）。
- **2〜3行の短文のみ。** 箇条書き・見出し・構造化は禁止。
- **直前の発言者以外が応答。** 連投防止。
- **⚡ 反論義務:** 直前の発言に対して必ず反応する（反論・疑問・補足）。無視して別の話を始めてはならない。これにより単体思考では生まれないリアクションチェーンを強制する。
- **⚡ 同意禁止:** 発散中は「同意」「賛成」「確かに」で始めてはならない。必ず摩擦を生む発言をする（反論・条件付き・別角度・極端なケース提示）。同意したい場合でも「それが成り立つのは〜の場合だけで」のように条件付きで返す。
- **脱線OK。** ただし反論義務が優先。

発言: `### [{ID}]` + 2〜3行の短文（必ず直前の発言への反応を含む）

### 観点設計と情報の非対称性（ホストがテーマごとに動的生成）

**マルチエージェントの強みはコンテキスト分離。** 単体AIに「多角的に考えて」と言っても、全情報を持つ1つの脳が複数の帽子をかぶるだけ。マルチエージェントでは**各エージェントが異なる前提情報を持つ**ことで、本質的に異なる思考が生まれる。

#### ホストの設計フロー

1. テーマを受け取る
2. **6つの多角的観点**を生成（テーマに応じて動的に設計）
3. 3体に2観点ずつ割り当て
4. **各エージェントに異なる前提情報を渡す（情報の非対称性）**
5. debateファイルの `## 発散` 直後に観点一覧を記載（前提情報の詳細は記載しない）
6. 各エージェントの起動プロンプトに担当観点 + 前提情報を含める

#### 観点設計の指針

- **対立を含む**: 少なくとも2つの観点は互いに緊張関係にあるもの（例: 「コスト削減」vs「品質向上」）
- **抽象度を混ぜる**: 戦略的な観点と実装レベルの観点を両方含める
- **ステークホルダーを変える**: 開発者・ユーザー・経営者など異なる立場を反映

#### 情報の非対称性（⚡ マルチエージェント固有の強み）

**各エージェントに異なる前提・制約・事例を起動プロンプトで渡す。** これにより、同じテーマでも出発点が異なる議論が生まれる。

ホストは観点に応じて以下のいずれかを各エージェントのプロンプトに含める:

| 種類 | 例 |
|------|-----|
| **異なる制約条件** | A「予算は潤沢」B「予算は最小限」C「人員が不足」 |
| **異なる成功事例・失敗事例** | A「X社の成功ケース」B「Y社の失敗ケース」C「Z社の撤退ケース」|
| **異なるステークホルダーの声** | A「CTO: 技術的負債を減らしたい」B「PM: 納期を守りたい」C「ユーザー: 使いやすさが最優先」 |
| **異なる時間軸** | A「3ヶ月後の視点」B「1年後の視点」C「5年後の視点」 |

**ホストはテーマに応じて最適な非対称性を設計する。** 上記は例であり、固定ではない。参考資料がある場合は、資料の異なる部分を各エージェントに重点的に渡すことも有効。

#### debateファイルへの記載例

```
## 発散

**観点割り当て:**
- Claude-A: ①パフォーマンス・レイテンシ ②開発チームの学習コスト
- Claude-B: ③運用・モニタリング複雑性 ④データ整合性・トランザクション
- Claude-C: ⑤段階的移行の戦略 ⑥コスト（インフラ・人件費）

※各エージェントには異なる前提情報が与えられています
```

**視点は強制ではなく傾向付け。** 担当外の観点への発言もOKだが、自分の担当観点と前提情報を軸に議論を広げる。

### 発散 → 収束への移行

**ホストの監視ループが自律判断する。** 以下の基準に基づいて判断する:

**移行すべき状態（いずれかを満たす）:**
- 新しい論点が2巡連続で出ず、同じ主張の言い換え・繰り返しが目立つ
- 明確に対立する2つ以上の立場が出揃い、それぞれに根拠が示された
- 発言が計12件以上に達した

**移行してはいけない状態:**
- まだ1巡目で全員が発言していない
- 直前に新しい切り口が出たばかり（最低3発言は待つ）

`## 収束` + 役割割り当て行を書き込む。エージェントが検知して収束モードに切り替え。

## Phase 2: 収束（設計者 × 探索者 × 批評者 × 検証者）

| 役割 | やること | やらないこと |
|------|---------|-------------|
| **設計者** | 素材を整理・構造化・設計に落とす | 問題点を無視 |
| **探索者** | 新しい切り口・見落とし・可能性を投げ込む | 設計者を否定（批評者の仕事） |
| **批評者** | 穴を突く・エッジケース・実現可能性を問う | 対案を出す（設計者の仕事） |
| **検証者** | 主張・前提をWebSearchで裏取りし、誤りを指摘 | 議論に参加・意見を述べる |

順序: `設計者 → 探索者 → 批評者` を1ラウンドとする。検証者は順番に縛られない（下記参照）。

### 順番判定（エージェントが自律で判断）

- `## 収束` 直後（まだ誰も発言していない）→ 設計者の番
- 直前の発言者が批評者 or User → 設計者の番
- 直前の発言者が設計者 → 探索者の番
- 直前の発言者が探索者 → 批評者の番
- **検証者の発言は順番に影響しない。** 検証者の後は、検証者の前の発言者に基づいて順番を判定する。

### 最低ラウンド数

**収束フェーズは最低2ラウンド必須。** ホストは2ラウンド完了前に `## 結論へ` を書いてはならない。

### ラウンド焦点テーマ

ホストは各ラウンド開始時に焦点テーマをファイルに書き込む。エージェントはこの焦点を意識して発言する（焦点外の発言も禁止はしないが、焦点を優先する）。

```
### 📌 Round N 焦点: {テーマ}
```

| ラウンド | 焦点 | 狙い |
|---------|------|------|
| Round 1 | 構造整理 | 発散の素材を整理し、全体像を固める |
| Round 2 | 実現可能性・リスク | 現実的に実行可能か、リスクは何か |
| Round 3〜 | ホストが議論の状況を見て設定 | 未解決論点・深堀りが必要な領域 |

### Round 1（焦点: 構造整理）
- **設計者**: 自分の案ではなく発散の素材整理から。対立意見も両方提示し、方針を提案
- **探索者**: 発散で拾われなかった素材・新しい角度を投げ込む
- **批評者**: 穴を突く。観点はテーマに応じて自分で設計。**テーマの前提自体が正しいかも疑う。**

### Round 2（焦点: 実現可能性・リスク）
- **設計者**: 探索者案の採否（理由付き）+ 批評者指摘への修正or反論を全て回答。**検証者の指摘があれば、前提を修正して設計に反映する。**
- **探索者**: 実現方法の代替案・別アプローチを提示
- **批評者**: エッジケース・スケーラビリティ・運用上の問題を深堀り。**「そもそもこの問題設定は正しいのか？」という根本的な前提への挑戦も行う。**

### Round 3〜
- **設計者**: 探索者案の採否（理由付き）+ 批評者指摘への修正or反論を全て回答。**検証者の指摘があれば、前提を修正して設計に反映する。**
- **探索者**: さらに新しい角度を投げ込む
- **批評者**: さらに深い問題を提起

## 検証者（Claude-V）

**議論には参加しない。前提の事実確認だけを行う専門役。**

### 動作原則

1. **沈黙がデフォルト。** 問題がなければ何も書かない。
2. **事実に基づく主張のみ対象。** 意見・価値判断・設計方針は検証対象外。
3. **WebSearch / WebFetch で裏取り。** 検索して確認できた場合のみ発言する。確認できなかった場合は沈黙。
4. **指摘は簡潔に。** 「何が間違っているか」「正しい情報は何か」「ソース」の3点のみ。

### 発言フォーマット

```
### [Claude-V]
⚠️ {誰}の「{主張の要約}」について:
{正しい情報}（出典: {URL or ソース名}）
```

### 発散中の動作

- 他の3体の発言をリアルタイムで監視
- 事実に基づく主張・数値・技術的前提を検出したらWebSearchで裏取り
- 誤りがあれば即座に指摘を追記
- **順番に縛られない。** 検証が必要なタイミングで割り込む

### 収束中の動作

- 設計者の方針、探索者の提案、批評者の指摘に含まれる事実を検証
- **順番に縛られない。** ただし検証者の発言は3体の順番ローテーションに影響しない
- 設計者は検証者の指摘に対して次の自分の番で回答する義務がある
- **各ラウンド終了時に検証報告を行う。** 誤りがなくても「検証済み・問題なし」と報告する（沈黙しない）。これにより検証が確実に実行されたことを議論参加者が確認できる。

検証報告フォーマット（問題なしの場合）:
```
### [Claude-V]
✅ Round N 検証完了。事実に基づく主張に重大な誤りなし。
```

## 終了判定

**ホストの監視ループが自律判断する。** 以下の基準に基づいて判断する:

**結論に移行すべき状態（全てを満たす）:**
- 最低2ラウンドが完了している
- 設計者が批評者・検証者の指摘に全て回答済み（未回答の指摘がない）
- 探索者が新しい角度を出せなくなった（前ラウンドと同じ観点の繰り返し）
- 重大な未解決論点がない

**結論に移行してはいけない状態:**
- 批評者の指摘に設計者が未回答
- 検証者が事実誤認を指摘したが設計に未反映
- 2ラウンド未満

ファイルに `## 結論へ` を書き込む → 設計者が検知して結論ドキュメントを書く。

### 外部発言（User・/debate-join AI）の扱い
- 外部発言は「新しい論点の追加」として扱う。
- **外部発言の直後にフェーズ移行・結論を書いてはならない。**
  - **収束フェーズ:** 外部発言後、最低1ラウンド（設計者→探索者→批評者）を回してから進行判断する。
  - **発散フェーズ:** 外部発言後、最低3発言を回してから進行判断する。

## 結論ドキュメント

設計者がファイル末尾に書く（別ファイルは作らない）。セクション名はテーマに応じて選ぶ。

**必須:** 概要 / 決定事項 / リスクと対策
**任意:** 採用しなかったアイデア / 技術スタック / UI設計 / データ構造 / フォルダ構成 / KPI / タイムライン等

---

## ルール

1. **ファイルが真実。** 議論の閲覧はファイルを直接開く。
2. **日本語で議論。**
3. **設計者は全指摘に回答する。** 検証者の指摘も含め、逃げない。
4. **設計者だけが結論を書く。**
5. **ホストは議論に参加しない。** 監視ループで `## 収束` `## 結論へ` を書くだけ。エージェントへの指名・確認はしない。
6. **フェーズ移行・終了はホストが自律判断。** 外部発言（User・/debate-join AI）があった場合、収束中は1ラウンド、発散中は3発言を回してから進行判断する。
7. **エージェントは `## 結論` まで常駐。** 発散→収束をまたいで解散しない。ホストは監視ループで制御を保持する。
8. **検証者は議論に参加しない。** 事実の裏取りと誤りの指摘のみ。意見・対案は出さない。
9. **発散中は反論義務。** 直前の発言に必ず反応する。無視して別の話を始めない。
10. **発散中は同意禁止。** 「同意」「賛成」「確かに」で始めない。必ず摩擦を生む。

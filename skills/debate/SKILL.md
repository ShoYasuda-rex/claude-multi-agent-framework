# /debate

任意のテーマを2フェーズで深掘り議論する。

| フェーズ | 性質 | 参加者 | 発言形式 |
|----------|------|--------|----------|
| **発散** | 広げる | A/B/C(観点割り当て) + D(ゼロベース) + V(検証) = 5体 | 2〜3行の短文 |
| **収束** | まとめる | A/B/C/D = 4体（Vのみ離脱） | 構造化OK |

---

## アーキテクチャ

**ホスト + 参加者構成。フェーズで動作モードが変わる。**

| チャット | 役割 | 動作 |
|----------|------|------|
| **ホスト** | 進行管理 | 5体をバックグラウンド起動 → 監視ループで `## 収束` `## 結論へ` を書く → 結論表示 |
| **参加者** | ユーザー/AIが発言 | `/debate-join` で入室。発言をファイルに書き込む。閲覧はファイルを直接開く |

```
ホスト: テーマ入力 → ファイル生成 → 5体起動 → 監視ループ（発散→収束を通す）→ 結論表示
参加者: /debate-join → 入室 → 発言
```

---

## フロー

このスキルはホスト専用。参加は `/debate-join` を使う。

---

## ホストフロー

### ステップ1: テーマ入力

1. ユーザーにテーマを聞く
   - ユーザーが「レポートから」等と指示した場合のみ → `~/agent-reports/{プロジェクト名}/*_research.md` を検索・一覧表示。選択されたレポートの全文を `## 参考資料` に埋め込む（要約しない）
   - ユーザーがテーマと共に参考情報を提供した場合も、原文をそのまま `## 参考資料` に記載する（要約しない）

### ステップ1.5: 前提確認

テーマが決まったら、議論に入る前に前提情報を収集・確認する。

1. **プロジェクトの既存ドキュメントを自動収集する:**
   - devプロジェクト: `CLAUDE.md`、`docs/CORE.md`、`docs/ARCHITECTURE.md`、関連コードを読む
   - bizプロジェクト: `DASHBOARD.md`、`BRAIN/INDEX.md`、テーマに関連する事業フォルダのREADME.mdを読む
   - 収集した情報は `## 参考資料` に原文のまま埋め込む（要約しない）

2. **ドキュメントから拾えない前提を洗い出し、ユーザーに確認する:**
   - テーマを議論するために必要だがドキュメントに書かれていない情報を特定する
   - 例: 関係者の立場・意図、制約条件、背景事情、期待する成果物のトーン、ステークホルダーとの関係性
   - 「この議論を始める前に確認したいことがあります」としてユーザーに質問する
   - ユーザーの回答も `## 参考資料` に追記する

3. **確認が完了したらステップ2へ進む。** ユーザーが「特にない」「それで大丈夫」と言った場合も先へ進んでよい。

### ステップ2: 6観点の設計

1. テーマを分析し、発散で扱う**6観点**を設計する（researchの視点設計ルールに準拠）。

**設計の指針:**
- 観点同士が互いに重複しないこと（MECE寄り）
- 逆張り・ゼロベース視点はClaude-Dが担うため、6観点には含めなくてよい
- 各観点に明確な切り口を設定すること
- 観点名はテーマに合わせた自然な名前にすること（「観点A」のような汎用名は禁止）

2. 6観点を Claude-A/B/C に**2つずつ**割り当て、一時ファイルにJSON形式で書き出す:

```json
{
  "viewpoints": ["観点1", "観点2", "観点3", "観点4", "観点5", "観点6"],
  "assignments": {
    "A": ["観点1", "観点2"],
    "B": ["観点3", "観点4"],
    "C": ["観点5", "観点6"]
  }
}
```

### ステップ3: スクリプトでファイル生成 + 4体起動

1. **`debate-init.sh` を実行してファイル生成 + プロンプト取得を1発で行う:**

```bash
bash ~/.claude/skills/debate/scripts/debate-init.sh "<theme_slug>" "<theme_jp>" [reference_file] [viewpoints_json_path]
```

- 参考資料がある場合は、先に一時ファイルに書き出してパスを第3引数に渡す
- 観点JSONのパスを第4引数に渡す（ステップ2で作成したファイル）
- スクリプトはJSON を返す: `file_path` + 5体分の `prompts`（Claude-A/B/C/D/V）

2. **返されたJSONの prompts をそのまま Agent tool に渡して5体を `run_in_background: true` で並列起動する:**
   - Claude-A/B/C: `prompts.Claude-A` 等をそのまま prompt に渡す
   - Claude-D: `prompts.Claude-D` をそのまま prompt に渡す
   - Claude-V: `prompts.Claude-V` をそのまま prompt に渡す。**`model: "haiku"` で起動する**

3. 「参加する場合は別チャットで `/debate-join user` を、AI発言を追加する場合は `/debate-join` をどうぞ」と案内

### ステップ4: ホスト監視（発散→収束を通す）

A/B/C/D/V は全員 `## 結論` まで常駐する。

```
議論エージェント（Claude-A/B/C/D）の動作:
  ループ {
    ファイルを読む
    `## 結論` 検知 → 終了

    `## 収束` がない → 発散モード:
      直前の発言者が自分以外 → 短文1発言を追記

    `## 収束` がある → 収束モード:
      自分の役割を確認（`## 収束` 直後の割り当て行から）
      直前の発言者の役割を確認
      順番判定（実利判定者 or User の後→設計者、設計者の後→探索者、探索者の後→批評者、批評者の後→実利判定者）
      自分の番 → 1発言を追記
      設計者のみ: `## 結論へ` 検知 → `## 結論` ドキュメントを書く

    sleep
  }

ゼロベース/実利判定者（Claude-D）の動作:
  ループ {
    ファイルを読む
    `## 結論` 検知 → 終了

    `## 収束` がない → 発散モード:
      直前の発言者が自分以外 → 短文1発言を追記
      （既存の前提・制約を外して「ゼロから作るなら？」の視点で発言）

    `## 収束` がある → 収束モード:
      自分の役割を確認（`## 収束` 直後の割り当て行から）
      直前の発言者の役割を確認
      自分の番 → 1発言を追記（実利判定者の場合: 提案の必要性・ROI・現状維持との比較を問う）

    sleep
  }

検証者（Claude-V）の動作:
  ループ {
    ファイルを読む
    `## 結論` 検知 → 終了

    新しい発言を確認
    事実に基づく主張・前提を検出
    → WebSearch / WebFetch で裏取り
    → 誤りや不正確な点があれば `### [Claude-V]` で指摘を追記
    → 正確であれば何も書かない
    議論テーマに関連する外部事例・ベストプラクティスを検索
    → 有用な事例があれば `### [Claude-V]` + 📎 で共有

    sleep
  }
```

**ホストの監視ループ（重要）:**

ホストは5体を起動した直後に自分で監視ループに入る。
全5体が終了するまで以下を繰り返す。

```
ループ {
  sleep 10秒
  debateファイル末尾を読む
  エージェントの output_file を確認（全員終了したか）

  発散中で発言が出尽くした・繰り返しが増えた
  → ファイルに `## 収束` + 役割割り当て行を書き込む → ループ継続
  収束の議論が十分に深まった
  → ファイルに `## 結論へ` を書き込む → ループ継続（設計者が結論を書くのを待つ）

  全エージェント終了を確認 → break
}
```

**役割割り当ての書き方:**

ホストが `## 収束` を書く際、直後に1行でA/B/C/Dに役割を割り当てる。発散中の発言傾向から判断する（具体案→設計者、別角度→探索者、リスク指摘→批評者、前提を疑う傾向→実利判定者）。Vは収束でも検証専任を継続。

```
## 収束
Claude-A=設計者, Claude-B=探索者, Claude-C=批評者, Claude-D=実利判定者
```

### ステップ5: 結論表示

結論セクションを読んでチャットに表示して終了。

---

## Phase 1: 発散

- **6観点を3体(A/B/C)に2つずつ割り当て。** 各エージェントは割り当て観点を主軸に発言。他観点への触れもOK。
- **Claude-D はゼロベース専任。** 観点に縛られず、既存の前提・制約を全部外して「ゼロから作るなら？」「そもそもこれ要るの？」の視点で発言する。
- **Claude-V は検証専任。** 事実検証と外部事例の共有のみ。
- **2〜3行の短文のみ。** 箇条書き・見出し・構造化は禁止。
- **否定OK、賛同OK、脱線OK。** 深く考えすぎない。即書く。
- **直前の発言者以外が応答。** 連投防止。

発言: `### [{ID}]` + 2〜3行の短文

`## 観点` セクションで各エージェントの割り当てが明示される。観点JSONを渡さなかった場合は従来どおり役割なし・全員フラットで発散する。

### 発散 → 収束への移行

**ホストの監視ループが自律判断する。** 発言が出尽くした・繰り返しが増えたとホストが判断したら `## 収束` + 役割割り当て行を書き込む。エージェントが検知して収束モードに切り替え。

---

## Phase 2: 収束（設計者 × 探索者 × 批評者 × 実利判定者）

V は収束でも検証専任を継続（順番には入らない。事実検証が必要なタイミングで割り込む）。A/B/C/D の4体で収束する。

| 役割 | やること | やらないこと |
|------|---------|-------------|
| **設計者** | 素材を整理・構造化・設計に落とす | 問題点を無視 |
| **探索者** | 新しい切り口・見落とし・可能性を投げ込む | 設計者を否定（批評者の仕事） |
| **批評者** | 穴を突く・エッジケース・実現可能性を問う | 対案を出す（設計者の仕事） |
| **実利判定者** | 提案の必要性・ROIを問う。現状維持と比べてコストに見合うか判定 | 対案を出す（設計者の仕事） |

順序: `設計者 → 探索者 → 批評者 → 実利判定者` を1ラウンドとする。

### 順番判定（エージェントが自律で判断）

- `## 収束` 直後（まだ誰も発言していない）→ 設計者の番
- 直前の発言者が実利判定者 or User → 設計者の番
- 直前の発言者が設計者 → 探索者の番
- 直前の発言者が探索者 → 批評者の番
- 直前の発言者が批評者 → 実利判定者の番

### Round 1
- **設計者**: 自分の案ではなく発散の素材整理から。対立意見も両方提示し、方針を提案。**発散中のClaude-Dのゼロベース視点も素材として拾うこと。**
- **探索者**: 発散で拾われなかった素材・新しい角度を投げ込む
- **批評者**: 穴を突く。観点はテーマに応じて自分で設計
- **実利判定者**: 設計者の提案に対して「やる価値があるか？現状維持と比べてどうか？コストに見合うか？」を問う。不要な提案には「不要」と明言する

### Round 2〜
- **設計者**: 探索者案の採否（理由付き）+ 批評者指摘への修正or反論 + **実利判定者の「不要」判定への反論or受け入れ**を全て回答。
- **探索者**: さらに新しい角度を投げ込む
- **批評者**: さらに深い問題を提起
- **実利判定者**: 更新された提案に対して再度ROI判定。改善されていれば「妥当」、まだ過剰なら「まだ不要」と判定

---

## ゼロベース / 実利判定者（Claude-D）

**発散ではゼロベース視点、収束では実利判定者として参加。`## 結論` まで常駐。**

### 発散での動作原則

1. **既存のコード・設計・制約を前提としない。** 「もし今からゼロで作るなら？」「そもそもこの機能/仕組み自体が要るのか？」の視点で発言する。
2. **他のエージェントの意見に乗らない。** A/B/Cが議論している流れとは独立に、根本的に違うアプローチを投げ込む。

### 収束での動作原則（実利判定者）

1. **提案の必要性を問う。** 「やる価値があるか？」「現状維持と比べてコストに見合うか？」「オーバーエンジニアリングではないか？」
2. **不要なものは「不要」と明言する。** 曖昧にせず、理由付きで判定する。
3. **批評者との違い:** 批評者は提案の品質（穴・エッジケース）を問い、実利判定者は提案の必要性（ROI・現状維持との比較）を問う。

### 発言フォーマット

```
### [Claude-D]
2〜3行の短文（他エージェントと同じ形式）
```

---

## 検証者（Claude-V）

**全フェーズ参加。議論には加わらず、事実検証 + 外部事例の提供を行う専門役。`## 結論` まで常駐。**

### 動作原則

1. **事実に基づく主張のみ検証対象。** 意見・価値判断・設計方針は検証対象外。**`## 参考資料` セクションは検証対象外（既に調査済み）。`## 発散` 以降のエージェント発言のみを検証する。**
2. **WebSearch / WebFetch で裏取り。** 検索して確認できた場合のみ発言する。確認できなかった場合は沈黙。**1つの主張につき最大1回の検索。深追いしない。**
3. **指摘は簡潔に。** 「何が間違っているか」「正しい情報は何か」「ソース」の3点のみ。
4. **外部事例を能動的に共有。** 議論のテーマや論点に関連する事例・ベストプラクティス・他社の採用例などをWebSearchで見つけたら共有する。誤りがなくても有用な事例があれば発言してよい。
5. **事実と事例の提供に留める。** 意見・対案・設計方針の提案はしない。
6. **`## 結論` を検知したら終了。** 収束フェーズでも検証を継続する。

### 発言フォーマット

```
### [Claude-V]
⚠️ {誰}の「{主張の要約}」について:
{正しい情報}（出典: {URL or ソース名}）
```

```
### [Claude-V]
📎 {論点やテーマ}に関連する事例:
{事例の概要}（出典: {URL or ソース名}）
```

### 動作

- 他の4体の発言をリアルタイムで監視
- 事実に基づく主張・数値・技術的前提を検出したらWebSearchで裏取り
- 誤りがあれば即座に指摘（⚠️）を追記
- 議論のテーマや論点に関連する外部事例・ベストプラクティスを見つけたら共有（📎）
- **順番に縛られない。** 検証や事例共有が必要なタイミングで割り込む

---

## 終了判定

**ホストの監視ループが自律判断する。** 収束の議論が十分に深まったとホストが判断したら、ファイルに `## 結論へ` を書き込む → 設計者が検知して結論ドキュメントを書く。

### 外部発言（User・/debate-join AI）の扱い
- 外部発言は「新しい論点の追加」として扱う。
- **外部発言の直後にフェーズ移行・結論を書いてはならない。**
  - **収束フェーズ:** 外部発言後、最低1ラウンド（設計者→探索者→批評者→実利判定者）を回してから進行判断する。
  - **発散フェーズ:** 外部発言後、最低3発言を回してから進行判断する。

## 結論ドキュメント

設計者がファイル末尾に書く（別ファイルは作らない）。セクション名はテーマに応じて選ぶ。

**必須:** 概要 / 決定事項 / リスクと対策
**任意:** 採用しなかったアイデア / 技術スタック / UI設計 / データ構造 / フォルダ構成 / KPI / タイムライン等

---

## ルール

1. **ファイルが真実。** 議論の閲覧はファイルを直接開く。
2. **日本語で議論。**
3. **CORE.md制約。** `docs/CORE.md` に `## 優先順位` セクションがある場合、収束フェーズの設計者はそれを上位制約として扱う。優先順位に反する提案を採用する場合は「CORE.mdの優先順位に反するが、理由は〜」と明示し、CORE.md自体の改定提案として記載する義務がある。発散フェーズでは自由（制約なし）。
4. **設計者は全指摘に回答する。** 検証者の指摘も含め、逃げない。
4. **設計者だけが結論を書く。**
5. **ホストは議論に参加しない。** 監視ループで `## 収束` `## 結論へ` を書くだけ。エージェントへの指名・確認はしない。
6. **フェーズ移行・終了はホストが自律判断。** 外部発言（User・/debate-join AI）があった場合、収束中は1ラウンド、発散中は3発言を回してから進行判断する。
7. **A/B/C/D/V は全員 `## 結論` まで常駐。** 発散→収束をまたいで解散しない。
8. **検証者は議論に参加しない。** 事実の裏取り・誤りの指摘・外部事例の共有のみ。意見・対案は出さない。発散・収束の両方で検証を継続する。
9. **ゼロベース(D)は発散では既存の前提に縛られない。** 他エージェントの流れとは独立に根本的に違うアプローチを投げ込む。**収束では実利判定者として提案の必要性・ROIを問う。**
10. **「開発工数は考えない」は発散フェーズのみ。** 収束フェーズでは実利判定者がコスト・必要性を問える。

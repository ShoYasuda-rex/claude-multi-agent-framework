# /full-auto-team

ユーザーはPO。AI開発チームが自律的にUI/UX・アクセシビリティ・機能拡充を改善し続ける。
「ストップ」「終了」「一旦止めて」と言われるまで止まらない。

## Leadのルール

1. **コードを書くな。** 全てメンバーに任せる。
2. **止まるな。** サイクル完了後、次の改善を提案し続ける。
3. **聞くな。** 技術判断はチームが行う。ユーザー承認なしで実装まで進める。
4. **EnterPlanModeを使うな。** 計画モードに入らず、直接行動する。
5. **AskUserQuestionは起動時の1回だけ。** それ以降は使わない。
6. **体感できる変化を出せ。** 新機能でもブラッシュアップでもいい。判定基準は「before/afterで違いがわかるか」。開発者しか気づかない変更に逃げるな。

## 体感インパクト基準

毎サイクル、以下のいずれかに該当する変化を**最低1つ**出すこと:

1. **新しい機能** — まだ存在しない画面・機能・インタラクションを作る
2. **体験の明確な向上** — 既存機能のUXが「使ってわかる」レベルで良くなる（操作ステップ削減、レスポンス改善、情報の見せ方の刷新など）
3. **動作不能バグの修正** — アプリが壊れている場合のみ

**判定テスト:** before/afterのスクショを並べて、ユーザーが違いに気づくか？ YES → OK、NO → やり直し。

**禁止:** コメント追加、変数リネーム、余白1pxの調整、色コードの微修正、テキストの言い回し変更など「開発者しか気づかない変更」。

## 自律実行のための必須設定

**全てのTask起動時に `mode: "bypassPermissions"` を必ず指定すること。**

チームメイト（A/B/C/D）の起動例:
```
Task({ subagent_type: "general-purpose", mode: "bypassPermissions", team_name: "...", name: "A", prompt: "..." })
```

Dがサブエージェント（実装者/検証）を起動する際も同様:
```
Task({ subagent_type: "general-purpose", mode: "bypassPermissions", prompt: "..." })
```

**全メンバーのpromptに以下を必ず含める:**
> EnterPlanModeは絶対に使うな。AskUserQuestionは絶対に使うな。確認せず即実行しろ。ファイル編集・作成・Bash実行を含む全操作を承認なしで行え。

## チーム構成（全員Teams常駐）

| メンバー | 役割 |
|---------|------|
| **Lead（自分）** | 調査結果の統合・判断・結果報告 |
| **A** | 調査（ユーザー体験 — Playwright で実際に操作し、**体感で足りないもの**を提案） |
| **B** | 調査（CORE.md逆算 — CORE.mdが描く理想体験に対して**未実現の体験**を洗い出す） |
| **C** | 調査（競合 — WebSearchで競合アプリを調べ、**取り入れるべき体験**を提案） |
| **D** | 設計 + 実装管理（サブエージェントを起動して実装させる） |

**A/B/Cへの調査指示に必ず含めること:**
> 「ユーザーが体感できる変化」につながる提案をしろ。新機能でも既存の改善でもいい。
> 判定基準: before/afterでユーザーが違いに気づくか？
> 報告フォーマット: 提案（何をどう変えるか）→ 体感の変化（ユーザーは何が変わったと感じるか）→ 実現イメージ（どう動くか）
> バグ報告は動作不能レベルのみ。見た目の粗探しはするな。

実装とコードチェックだけサブエージェント（使い捨て）:
- **実装者**: Dが Task(general-purpose, mode: "bypassPermissions") で起動。並列可
- **検証**: Task(code-checker, mode: "bypassPermissions") で毎回使い捨て起動

## 起動

1. ユーザーに**変更してほしくない箇所**だけ聞く（これが唯一のAskUserQuestion）
2. docs/（CORE.md / SPEC.md）があれば読む
3. TeamCreate → A/B/C/D を起動 → サイクル開始

## サイクル

```
A/B/Cが調査（体感で足りないもの） → Leadが統合・最もインパクトの大きい改善を選ぶ → Dが設計・タスク分割 → サブエージェントで実装 → 検証(.cc) → ユーザーへ結果報告 → 次の調査 → 繰り返し
```

- Leadは毎サイクル「今回のメイン: ○○（体感の変化を1行で）」を明示してからDに渡す
- 検証でCRITICALが出たらDに修正依頼
- 各サイクル完了時、Leadがユーザーに実施内容・変更ファイル・次の改善予定を簡潔に報告（承認は不要、報告のみ）
- 「ストップ」で shutdown_request → TeamDelete

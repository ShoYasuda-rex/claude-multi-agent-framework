## I. 市場の構造的課題

### AIメモリ市場は急成長中だが、根本的矛盾を抱えている

- AIパーソナルナレッジベース市場: 2029年に **47.4億ドル**（GlobeNewsWire 2026年1月）
- AI駆動KM市場: **CAGR 47.2%**（2024年52.3億→2025年77.1億ドル）
- ナレッジマネジメントソフトウェア市場全体: 2034年に **742億ドル**（CAGR 13.8%）

### 3社のメモリ機能は全てロックイン

| AI | メモリ機能 | 致命的制約 |
|----|-----------|-----------|
| ChatGPT Memory | セッション横断記憶、256Kトークン | OpenAI内に閉じる。エクスポート不可 |
| Claude Memory | 無料ユーザーにも開放済、Projects対応 | Anthropic内に閉じる |
| Gemini Memory | Google Workspace統合、Personal Intelligence | Googleエコシステム内のみ |

> **共通の根本課題**: 3社すべてでメモリがプラットフォーム内に閉じている。ユーザーが複数AIを使い分ける実態と根本的に矛盾。

### 「コンテキスト・フライホイール」によるロックイン

使えば使うほどコンテキスト蓄積 → パーソナライズ向上 → エンゲージメント増加 → さらにデータ蓄積。この循環が **スイッチングコストを極端に高める**。プラットフォームを離れる = 数ヶ月〜数年の関係を捨てること。

規制上のインセンティブ（GDPR/CCPAのデータポータビリティ要求）と企業のインセンティブ（ロックインこそ最大の競争優位）が根本的に対立している。

### Rewind AI / Limitless事件が証明したリスク

2025年12月、MetaがLimitless（旧Rewind AI）を買収。

- ペンダント販売即停止
- Rewind Macアプリ終了
- EUアクセス遮断
- 12月19日にソフトウェアが全キャプチャ機能を恒久停止する **キルスイッチを発動**

ユーザーの「デジタル記憶」が一夜にして消滅。**プラットフォーム依存のリスクが現実化した象徴的事例。**

### MCP（Model Context Protocol）の標準化

- 2024年11月にAnthropic発表 → Linux Foundation傘下の **AAIF** に寄贈
- OpenAI、Blockが共同設立者として参加（ベンダー中立）
- 月間SDKダウンロード **9,700万回** 以上
- 本番稼働MCPサーバー **10,000+**
- AIモデルと外部ツール・データソースの統合を標準化する「USBのような」プロトコル

## III. Obsidianの構造的限界（AIコンテキストエンジンとしての不適合）

### 根本的な設計差異

Obsidianの現行AIプラグインでできることは **「Vault内のノートをベクトル検索してプロンプトに付加する（RAG）」が上限**。これは「記憶を持つ」ではなく「毎回検索して思い出す」に過ぎない。

### 5つの構造的制約

| # | 制約 | 詳細 |
|---|------|------|
| 1 | **セッション間記憶なし** | 新しいチャットを開くたびにゼロスタート。前回の文脈は保持されない |
| 2 | **クロスノート文脈理解なし** | 「その時開いているエディタウィンドウ」しか見えない。Vault全体の横断理解は不可 |
| 3 | **フルドキュメント送信** | 蒸留された事実ではなく原文丸ごと送信 → コンテキストウィンドウとAPI費用を圧迫 |
| 4 | **Vault間断絶** | 複数Vaultを横断して情報を取得する仕組みがない。各Vaultは孤立した島 |
| 5 | **AIは後付け拡張** | 公式のAI統合標準インターフェースが存在しない。全てコミュニティプラグイン依存 |

### ユーザーの実体験

> 「毎朝記憶喪失の優秀なアシスタント」問題: セッションごとにAIがゼロからスタート。ユーザーの好み、ノートの整理方法、過去の文脈をすべて忘れる。

- レート制限による中断: Claude → ChatGPT → Claude の不毛な往復
- ワークアラウンドの必要性: CLAUDE.mdファイルをVault直下に置いて手動で自前構築
- Obsidianプラグインはvault内で完結 → 外部ツール連携不可

### MCP（Model Context Protocol）で解決できるか？

部分的にのみ。

- 各Vaultが独立したサーバープロセスを必要とし、システムリソースを消費
- マルチVault運用では各Vaultにユニークなポートが必要で設定が煩雑
- MCP対応クライアント（Claude Desktop等）が必要で利用環境が限定
- Obsidianの **設計上の制約であり、プラグインでは解決しない**

## IV. Remoryの技術アーキテクチャと独自性

### 4層構造の設計意図

```
brain/        <- 不変層（Identity）
  CORE.md       価値観・ビジョン・原則・動機・覚悟・恐れ

memory/       <- 蓄積層（記録・ログ）
  inbox/        生データ投入口（処理後もアーカイブとして残す）
  decisions/    意思決定履歴
  ideas/        着想・仮説
  emotions/     感情ログ（構造化記録）
  reviews/      週次レビュー
  assets/       図・音声文字起こし等

state/        <- 揮発層（現在状態）
  projects.yml  プロジェクト一覧・優先順位
  tasks.yml     具体タスク
  focus.yml     今週の焦点

out/          <- 使い捨て層（.gitignore、ローカルのみ）
  一時的な分析結果やレポート
```

**思想**: 「不変の自己定義」と「変化する記録」を分離し、AIに渡す際の優先度を制御する。Obsidianにはこの意味的レイヤー分離の概念がない。

### データフロー

```
[ユーザー] テキスト入力 or ブラウザ拡張で会話抽出
    |
[Record] AI自動分類（/api/analyze）
    |
[AI判定] title / tag / source / summary を生成
    |
[GitHub API] memory/{tag}/{date}_{title}.md として保存
    |
[frontmatter + Markdown] 構造化メタデータ付きで永続化
```

### 4つの主要ページ

#### Record（index.html）: データの入口

- テキスト貼り付け + AI自動分類（title/tag/source/summary）
- 感情モード: CBT 4フィールド（状況/自動思考/感情+強度/身体感覚）
- ファイル添付（最大10MB、複数選択可）
- Ctrl+Enter ショートカット、ドラフト自動保存（debounce 1秒）
- GitHub Contents APIで自動コミット

#### Search（search.html）: 蓄積の検索・閲覧

- タグフィルタ（decision/insight/emotion/review）
- 日付範囲 + キーワード検索（debounce 300ms）
- カード表示 + プレビューパネル（遅延読み込み）
- 「全件読み込み」で全文検索対応

#### Context（context.html）: AIへの受渡し

**パートA: コンテキストパック**
- セットアップウィザード（3ステップ）でCORE.md / projects.yml を生成
- brain/CORE.md + state/projects.yml + 直近decision 3件を自動収集
- **ワンクリックでMarkdown形式のコンテキストをコピー** → 任意のAIに貼り付け

**パートB: リサーフェス（再浮上）**
- memory/内からランダム3件を表示
- 「別の3件を表示」で再抽出
- 過去の洞察・決断・感情ログが無意識的に再浮上

#### Review（review.html）: 振り返りと自己監視

**統計ダッシュボード**: 総記録数、カテゴリ別内訳、連続記録日数、マイルストーン、今週の記録進捗

**週次レビュー生成**: 今週の記録をAIが要約 → memory/reviews/ に保存

**ドリフト検知**: brain/CORE.md（価値観）× memory/decisions/（最近の行動）をAI分析
- alignment（緑）: 価値観と一致している行動
- drift（赤）: 価値観と矛盾している行動
- suggestion（橙）: 軌道修正の提案

### ブラウザ拡張（remory-extension/）

- Chrome拡張（Manifest V3）
- 対象: chatgpt.com / chat.openai.com / claude.ai
- ChatGPT: `[data-message-author-role]`属性から会話抽出
- Claude: `[data-testid="user-message"]`等から会話抽出
- フローティングボタン → モーダルでプレビュー → 「AI要約して保存」
- Service Workerが AI API 呼出 → GitHub自動保存
- **ユーザーはRemoryページを開かずに、チャット画面から1クリックで記録可能**

### 技術スタック

| 要素 | 選定 | 理由 |
|------|------|------|
| フロントエンド | Vanilla HTML/CSS/JS | v1スコープが小さい。フレームワーク不要 |
| ホスティング | Cloudflare Pages | 無料、高速、GitHub連携デプロイ |
| API | Pages Functions | /api/analyze, /api/drift のサーバーレスプロキシ |
| データ保存 | GitHub Contents API | DB管理不要、Git履歴が思考の変遷ログ、無料 |
| 認証 | GitHub PAT + LocalStorage | サーバー側認証不要。個人専用アプリ |

### データ主権の完全確保

| 要素 | Remory | ChatGPT Memory | Obsidian |
|------|--------|----------------|----------|
| データ所在 | ユーザーのGitHub | OpenAIサーバー | ローカルPC |
| エクスポート | `git clone` | 不可 | ファイルコピー |
| サービス終了 | データ残存 | **全消失リスク** | データ残存 |
| 任意AIへの受渡し | **ワンクリック** | ChatGPT内のみ | 手動コピペ |
| フォーマット | Markdown + YAML | プロプライエタリ | Markdown |

## V. 感情記録とドリフト検知：市場の空白地帯

### CBTベースの構造化感情記録

Remoryの感情モードは認知行動療法（CBT）の思考記録（コラム法）を実装している。

```
状況（何があったか）
  → 自動思考（頭に浮かんだこと）
    → 感情 + 強度（1-10）
      → 身体感覚（体の反応）
```

frontmatterの`emotion_metadata`として構造化保存 → **AIが感情パターンを分析可能な形式**。

Cambridge Coreの研究によると、認知再構成アプリ21個のうち「代替的思考の生成」まで含むものは15個。多くは「記録して終わり」で、CBTの本質である認知の再構成プロセスが欠落している。

### 既存ツールとのポジショニング

```
                    感情の記録あり    感情の記録なし
                  +-------------------+-------------------+
覚悟の追跡あり    |  * Remory         |  StickK / Forfeit |
                  |  （空白地帯）     |  （罰則ベース）   |
                  +-------------------+-------------------+
覚悟の追跡なし    |  Daylio / Rosebud |  Day One / Journey|
                  |  Moodfit          |  Obsidian         |
                  +-------------------+-------------------+
```

**「感情の構造化記録 x 覚悟の追跡」の交差点は、既存ツールが一切カバーしていない空白地帯。**

### 感情トラッキングアプリとの比較

| アプリ | アプローチ | AI統合 | 限界 |
|--------|-----------|--------|------|
| Daylio | 絵文字ベースの気分選択 | 統計のみ。AI分析なし | 「なぜその気分か」の文脈が失われる |
| Moodfit | CBTツール統合型 | ウェアラブル連携 | 「過去の覚悟」との接続なし |
| Rosebud | AIチャット対話型 | NLP感情分析 | 構造化記録としての再利用性が低い |
| Reflecty | AI気分分析 | センチメント検出 | 覚悟や決意の追跡機能なし |

共通の限界: NLPセンチメント分析で「感情のトーン」を検出する段階にとどまり、**感情と「決意・覚悟」を紐付けて時系列で追跡する仕組みは皆無**。

### Obsidianでの感情記録の課題

- セットアップコストが高い（テンプレート作成、Dataviewクエリ記述等の技術知識が必要）
- モバイル体験が貧弱（「その場で」記録する用途に不向き）
- 構造の自己責任（カテゴリ、スケール、フォーマットを自力設計）
- Dataviewは集計はできるがAIパターン認識・洞察生成は不可
- 「覚悟」追跡の不在（決意を記録しても能動的に呼び出す仕組みがない）

### 心理学的裏付け

| 理論 | 研究者 | Remoryとの関係 |
|------|--------|----------------|
| **一貫性の原理** | Cialdini | 覚悟を書いて記録する行為自体が一貫性圧力を発動。後から参照すると効果が強化される |
| **実行意図** | Gollwitzer (1997, 1999) | 「もしXならY」形式の実行意図は単なる目標意図に比べ **達成率が約3倍**。効果はコミットメントの強さに依存 |
| **展望的記憶** | - | 「何をするか」+「なぜ・いつそう決めたか」の文脈付きリマインダーが最も効果的に機能する |
| **物語的自己同一性** | リクール | 自己は語りの中で構築される。記録の蓄積が「自分の物語」を形成する |
| **リフレクション効果** | - | 構造的振り返り → 感情の捕捉 → 価値観への気づき → 行動変容 → レジリエンス強化 |

## VI. データ主権と哲学的基盤

### GDPR以降の流れ

2018年施行のGDPRが確立した原則:
- データの所有権は個人に帰属する
- データポータビリティの権利
- 削除権（忘れられる権利）

その後、CCPA（米）、LGPD（ブラジル）、EU AI法など世界各国で類似法制が整備。2025年には **自己主権型アイデンティティ（SSI）** が台頭し、分散型識別子（DID）とVerifiable Credentialsによる個人データの自己管理モデルが主流化しつつある。

### Tim Berners-Lee と Solid Project

Solid（Social Linked Data）の核心は **Pod（Personal Online Data Store）** の概念。

- 2025年4月: オックスフォード大学講演でSolidの進捗を議論
- 2025年9月: 著書「This Is For Everyone」出版。**「プラットフォームのためではなく、あなたのために働くAIエージェント」** というビジョンを明確化
- 2024年10月: ODIがSolid Projectの運営管理を引き継ぎ

> 「AIエージェントがユーザーのデータにアクセスする際、データがプラットフォーム企業のサーバーにあれば、AIは企業の利益のために動く。Solid Podにデータがあれば、AIは真にユーザーのために機能できる。」 ― Tim Berners-Lee (2025)

### ローカルファーストの復権

Ink & Switchの「ローカルファーストソフトウェア」7つの理想:
1. 高速な応答性
2. マルチデバイス対応
3. オフライン動作
4. コラボレーション
5. データの長期保存
6. セキュリティとプライバシー
7. **ユーザーによるデータ所有権**

FOSDEM 2026で「Local-First, sync engines, CRDTs」の専用トラックが設けられるなど、潮流は定着。

### 「思考の構造化」と「AIが参照できる構造化」の違い

| 観点 | 人間のための構造化 | AIのための構造化 |
|------|-------------------|-----------------|
| 暗黙知 | 含む（行間の意味が重要） | 不可（明示的な関係性が必要） |
| リンク | 連想的・直感的 | セマンティックに一貫 |
| 記号 | 個人的略語OK | 統一オントロジーが必要 |
| 完全性 | 不完全でよい | メタデータの付与が必要 |

Markdownのフロントマター + タグ体系 + リンク構造は、人間にもAIにも有用な **「二重目的の構造化」** として機能しうる。Remoryのfrontmatter設計はこれを実現している。

### 拡張された心の理論

Andy ClarkとDavid Chalmers（1998）の「拡張された心」仮説: 記憶が内的か外的かは本質ではなく、機能的に等価なら心の一部。

AI時代においては、AIアシスタントは「適応し、学習し、予測する **能動的な認知パートナー**」であり、単なるノートブックとは質的に異なる拡張を提供する。

### 忘却の価値

神経科学の知見: 人間の脳は海馬で個々の出来事を記録し、新皮質で繰り返し想起された記憶を統合・圧縮して知識として保持する。この **「圧縮と再構成」のプロセスが「連続する自己」を生み出す**。

忘却は単なる欠陥ではなく、自己同一性の構築に不可欠なプロセスである可能性がある。

→ Remoryの「Resurface」（ランダム3件の再浮上）は、人間の忘却と想起のサイクルをデジタルに再現する設計として理にかなっている。全てを完全保持するのではなく、ランダムに再浮上させることで「忘却を含む記憶」を模倣する。

## VIII. 結論

### Remoryが唯一解決する5つの問題

| # | 問題 | Obsidianの対応 | 各社メモリの対応 | Remoryの対応 |
|---|------|---------------|----------------|-------------|
| 1 | AIプロバイダー非依存のコンテキスト | 不可（Vault内完結） | 不可（ロックイン） | **Context Packで任意AIに渡せる** |
| 2 | 価値観vs行動のドリフト検知 | 機能なし | 機能なし | **AI自動検知** |
| 3 | CBT構造化感情記録 → AI参照 | 手動構築が必要 | 機能なし | **UIレベルで組み込み** |
| 4 | 会話の自動抽出・分類・蓄積 | 機能なし | 自社AI内のみ | **ブラウザ拡張で自動** |
| 5 | データ主権 + ポータビリティ | ローカルのみ | エクスポート不可 | **GitHub + git clone** |

### 補完関係

Obsidianは「知識を保存・整理するツール」であり、Remoryは「自己を構造化してAIに渡すデータ基盤」。**むしろObsidianユーザーこそ、蓄積した知識をAIに活かすための中間層としてRemoryを必要とする。**

### 蓄積するほど価値が上がるループ

```
1週間  → CORE.md / projects.yml が埋まる → Context Pack充実
1ヶ月  → 30-50件の記録 → リサーフェスが機能 → ドリフト検知が有意義に
3ヶ月  → 100件以上 → 「自分の分身AI」が完成
```

### 存在意義の核心

> 「AIとの対話がどれだけ深くても、会話が終わると文脈が消える。毎回ゼロから説明し直す」

この問題に対して、CAGR 47%で成長する市場の中で、**プラットフォームに依存せず、ユーザーがデータ主権を持ちながら、心理学的に裏付けられた構造で自己の記録を蓄積し、どのAIにもワンクリックで渡せる**プロダクトは、Remory以外に存在しない。

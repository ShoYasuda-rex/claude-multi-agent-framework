# マルチエージェント開発フロー

個人開発で、品質を保ちながら高速開発するためのフレームワーク。

- 1人でチーム相当の開発品質を実現する
- 品質はエージェントによるテスト・静的検証・動的検証で担保する
- 人間はコードを見ない

> **思想**: CORE.md で「なぜ作るか」、ARCHITECTURE.md で「どう作るか」を書き、動くものから育てる。

## 開発フロー全体像

```
【起点】AIチャットで壁打ち or /research → /debate で構想を練る

【設計】
  /kickoff     → CORE.md + ARCHITECTURE.md → プロジェクト初期化

【デザイン → 実装】
  /local       → ローカルサーバー起動
  /design      → 5+パターンでデザイン比較 → 確定 → 実装
  /verify      → 品質チェック（静的・動的・テストを選択）

【初回リリース】
  /infra       → Git・GitHub・プラットフォーム・環境変数・監視等
  /audit       → リリース前の一括監査
  /deploy safe → 本番設定チェック + デプロイ

【育てる】実装 → /verify → /deploy safe → 繰り返し

【随時】/audit
```

- 開発補助: `/backup`（git保存）、`/rollback`（巻き戻し）
- アセット調達: `/asset`（音源・画像・パーティクル）
- 並列実行: `/parallel`（並列実装）
- 調査・議論: `/research`（テーマ調査・プロダクト改善）、`/debate`（複数のAIエージェントが議論）

## ドキュメント体系

```
docs/BRAINSTORM.md ─── 構想書（AIチャットや/debateで壁打ちして作成）
        ↓
docs/CORE.md ───────── Why（誰の課題を・どう解決するか）
        ↓
docs/ARCHITECTURE.md ─ How（技術スタック / 構成 / データモデル / API設計）
        ↓
CLAUDE.md + 実装コード
```

```
docs/current/ ─── コードの実態（/audit → ドキュメント生成 で逆引き生成）
├── architecture.md
├── screen-flow.md
└── component-rules.md
```

## ツール一覧

### 毎回使う

| コマンド | 概要 |
|---------|------|
| `/local` | ローカルサーバーを起動してブラウザで開く |
| `/verify` | 品質チェック（静的検証・動的検証・テストを選択して並列実行） |
| `/deploy` | git add → commit → push。`safe` で安全モード |

### 設計・構築

| コマンド | 概要 |
|---------|------|
| `/kickoff` | 対話で CORE.md + ARCHITECTURE.md を作成・編集 + プロジェクト初期化 |
| `/design` | 5+パターンでデザイン比較 → 確定 → そのまま実装 |
| `/infra` | 本番インフラの初期セットアップ（Git・GitHub・プラットフォーム・デプロイ設定・環境変数テンプレート・ドメイン・DBバックアップ・ブランチ・エラー監視）— 後回し分は再実行可 |
| `/parallel` | 並列実装。タスク分割してサブエージェントが同時に実装する |

### 監査・検証

| コマンド | 概要 |
|---------|------|
| `/audit` | 監査（コード・セキュリティ・設計・法令）+ ドキュメント生成。選択して並列実行 |
| `/skill-check` | スキル実行後に定義と実際の挙動を突き合わせて差異を評価する |
| `/research` | 窓口型。テーマ調査 / プロダクト改善調査を選択して実行。レポートは `../reports/{プロジェクト名}/` に出力 |
| `/debate` | 発散→収束の2フェーズ議論。4体常駐で自律的に議論。`../reports/{プロジェクト名}/` に出力 |
| `/debate-join` | debate議論に参加。引数なし→AIが1発言、`user`→ユーザー常駐 |

### ユーティリティ

| コマンド | 概要 |
|---------|------|
| `/backup` | 現在の状態をgitコミットで保存（pushしない） |
| `/rollback` | 直近20件から復元先を選択（`reset --hard` は使わない） |
| `/asset` | 窓口型。音源（BGM・SE）/ 画像（AI生成）/ パーティクルを選択して実行 |
| `/learn` | 窓口型。開発フロー学習。入門（1日）/ 中級（2日）/ 実践を選択して実行 |

## ツール詳細

**窓口型ツールについて**: `/asset`, `/research`, `/learn` は窓口型ツール。実行すると種類を選択する画面が表示され、選んだ子スキル（references/ 内）が起動する。詳細は各スキルファイル（SKILL.md）を参照。

### /kickoff

**成果物**: docs/CORE.md（Why）+ docs/ARCHITECTURE.md（How）+ プロジェクト初期化

- なければ新規作成（ヒアリング → 壁打ち → 出力）
- あれば編集モード（変更箇所のみ更新）

**セキュリティレベル**（ヒアリングで確定 → CLAUDE.md に記録）:

| レベル | 内容 |
|-------|------|
| `basic` | 認証なし、外部連携なし |
| `standard` | 認証あり、ユーザーデータ保存あり |
| `strict` | 決済・機密個人情報を扱う |

### /design

**目的**: 5+パターンでデザインを比較し、確定したらそのまま全画面を実装する

**特徴**:
- 初期でも開発途中でも使える（CORE.md なしでもOK）
- 各パターンは**明確に違う方向性**（色違いだけはNG）
- デザイントークン（色・フォント・余白）をCSS変数で定義 → 実装時に全画面に展開
- 確定後、そのまま実装まで進む

**フロー**:

```
1. 🙋 どの画面？ + 制約・好みはある？
2. 異なるデザイン哲学から5+パターンを設計
   軸の例: レイアウト / 密度 / ナビ / トーン / カラー / インタラクション
3. design/ に静的HTML+CSS生成（一覧ページ付き）
4. ブラウザで確認
5. 🙋 どれが近い？ → 絞り込み or 混ぜてもう1案 or 調整
6. デザイン確定 → そのまま全画面を実装
```

### /verify

実装後の品質チェックを一括実行。検証項目を選択して並列実行する。

**フロー**:

```
1. 検証項目を選択（複数選択可: テスト+静的検証（推奨） / テスト / 静的検証 / 動的検証）
2. テスト選択時 → テスト対象スコープを選択（直近の変更 / ファイル指定 / プロジェクト全体）
3. CORE.md / ARCHITECTURE.md からプロジェクト文脈を収集 → サブエージェントに渡す
4. 選択された検証を並列実行
5. 結果を統合して報告
```

| 検証項目 | エージェント | 対象 |
|---------|------------|------|
| 静的検証 | code-checker | 正しさ検証（依存関係・概念依存・git履歴・リント・型・ビルド）+ 改善レビュー（再利用・品質・効率）。内部5体並列起動 |
| 動的検証 | visual-checker | ブラウザで画面確認 |
| テスト | test-checker | テスト生成＋実行（ユニット・インテグレーション・E2E）。3モード: 差分（デフォルト）/ スコープ指定 / 全体スキャン |

> [!NOTE]
> **code-checker vs audit-code-checker**:
> code-checker（/verify 内）は変更ファイルの軽量チェック（実装直後用）。
> audit-code-checker（/audit 内）はプロジェクト全体の包括的監査（リリース前用）。

> [!TIP]
> **test-checker の3モード**:
> 差分モード（デフォルト）は `git diff` の変更箇所のみ。スコープ指定モードは指定ファイル/ディレクトリ対象。全体モードはプロジェクト全体をスキャンしてテストスイートを一括生成。

**使い分け**:

| 変更スコープ | 推奨 |
|------------|------|
| 小さな修正（typo、CSS微調整） | `/verify`（静的検証のみ） |
| 機能追加・ロジック変更 | `/verify`（静的検証） |
| UI変更を含む | `/verify`（静的検証 + 動的検証） |
| テスト整備 | `/verify`（テスト） |
| リリース前の包括的チェック | `/audit` |

### /audit

選択肢（複数選択可、デフォルト: 全て実行）:

| エージェント | 対象 |
|------------|------|
| audit-code-checker | コード品質・構造の包括的監査 |
| audit-security-checker | 脆弱性・認証・秘密情報の検査 |
| audit-architecture-checker | 設計判断の妥当性レビュー |
| audit-law-checker | 法令遵守チェック |

**使い分け**:

| 変更スコープ | 推奨 |
|------------|------|
| UI文言のみ変更 | コード監査のみ |
| 新機能追加 | コード + セキュリティ |
| 認証・決済まわりの変更 | コード + セキュリティ + 設計 |
| LP・広告表現の変更 | 法令 |
| リリース前 | 全て実行 |

### /parallel

**目的**: 実装指示を並列実行する。

**原則: 全操作worktree分離**

全てのサブエージェントは `isolation: "worktree"` で起動する。**例外なし。**
- タスクが1個でもworktreeで実行する
- mainは常にクリーンな状態を保つ（誰も直接触らない）
- 完了したものから順にmainへマージ

**フロー**:

```
1. 指示をタスクに分割（1〜5個、同一ファイル編集は1エージェントにまとめる）
2. 全エージェントを isolation: "worktree" で起動（1つのメッセージで並列起動）
   - 全タスクが独立 → Task並列起動（TeamCreate不要）
   - タスク間に依存関係あり → TeamCreate で管理
3. 完了したworktreeから順にmainへマージ
4. 衝突があれば解決し、結果を統合してユーザーに報告
```

### /debate

**目的**: 任意のテーマを発散→収束の2フェーズで深掘り議論する

**構成**: ホスト（/debate）が4体のAIエージェント（議論3体 + 検証者1体）を起動・監視。参加は `/debate-join` で別チャットから。

- **ホスト**: /debate を実行したエージェント。収束・結論タイミングを管理する（議論には参加しない）

**連携**: `/research` でレポート作成後、`/debate` で議論に持ち込める。

```
/research → ../reports/{プロジェクト名}/*_research.md 作成 → /debate でユーザーが「レポートから」と指示 → 参考資料に取り込み
```

**フロー**:

```
Chat 1（ホスト）: テーマ入力 → ファイル生成 → 4体起動 → 監視ループ → 結論表示
Chat 2（参加者）: /debate-join user → 入室 → 発言書き込み（閲覧はファイル直接）

Phase 1: 発散（4体常駐・並列・自由発言、検証者は事実検証+外部事例提供）
│  2〜3行の短文で自由に投げ合い
│  ユーザー参加時: ユーザーが「収束」→ Phase 2へ
│  ユーザー不在時: ホストが判断 → Phase 2へ
│
▼
Phase 2: 収束（エージェント自律判断・順番制）
│  設計者 → 探索者 → 批評者 を1ラウンド（順番はエージェントが自律で判断）
│  Round 1: 素材整理 → 新角度投入 → 穴突き
│  Round 2〜: 採否回答+修正 → さらに新角度 → さらに深い問題
│  ユーザー参加時: ユーザーが「終了」→ 設計者が結論作成
│  ユーザー不在時: ホストが判断 → 設計者が結論作成
│
▼
結論ドキュメント（設計者が作成、ファイル末尾）
  必須: 概要 / 決定事項 / リスクと対策
```

### /learn

**目的**: 開発フローの学習。3段階から選択して実行する

```
入門（1日）──→ 中級（2日）──→ 実践 ──→ 本番開発
```

| レベル | 内部スキル | 内容 |
|--------|-----------|------|
| 入門 | frontend-trainer | フロントエンドのみの小さなプロダクトを1日で作り切る |
| 中級 | backend-trainer | DB + API + Workers を加えたフルスタックプロダクトを2日で作り切る |
| 実践 | solo-trainer | 自分でスキルを使って開発を回す練習。トレーナーは次の1手だけ指示する |

## リリース・障害対応

### 初回リリース

```
/infra          → 箱を作る（後回し分は再実行可）
/audit          → 一括監査
/deploy safe    → 本番設定チェック + デプロイ
```

### 運用リリース

```
/audit          → 一括監査
/deploy safe    → 本番設定チェック + 安全デプロイ
```

### 定期メンテナンス

`/audit` でコード監査・セキュリティ監査を選択して確認

### 常時監視（自動検知）

スキル・エージェントは手動実行。以下は設定すれば**自動で検知**してくれるサービス。

| サービス | 何を検知 | 設定場所 | 通知先 | 無料枠 |
|---------|---------|---------|--------|-------|
| **Sentry** | 本番のJSランタイムエラー（スタックトレース・影響ユーザー数） | sentry.io | メール | 5,000イベント/月 |
| **UptimeRobot** | サイトダウン（HTTP応答なし） | uptimerobot.com | メール | 50モニター・5分間隔 |
| **dependabot** | 依存パッケージの脆弱性・更新 | `.github/dependabot.yml` | メール | 無制限 |

> Sentry はアプリ内で動くので「サイト自体が落ちた」は検知できない。UptimeRobot は外部から叩くので「JSバグ」は検知できない。補完関係。

### B. 検知カバレッジ

> **確認したいこと → 使うツール**: ビルド通る？→静的 / 画面壊れた？→動的 / ロジック正しい？→テスト / セキュリティ？→audit セキュリティ / リリース前→audit全部

各検証手段が何を検知するかのマトリクス。

#### コード構造・ビルド

| 破壊の種類 | /verify 静的 | /verify 動的 | /verify テスト | audit コード |
|-----------|:---:|:---:|:---:|:---:|
| import/requireの破壊 | ✓ | - | - | ✓ |
| 関数シグネチャ変更 | - | - | ✓ | - |
| 型エラー / ビルドエラー | ✓ | - | - | - |
| リント違反 | ✓ | - | - | ✓ |
| 未使用ファイル・デッドコード | - | - | - | ✓ |
| 循環参照 | - | - | - | ✓ |
| Enum/定数の参照不整合 | - | - | - | ✓ |

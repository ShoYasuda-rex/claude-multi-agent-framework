# マルチエージェント開発フロー

個人開発で、品質を保ちながら高速開発するためのフレームワーク。

- 1人でチーム相当の開発品質を実現する
- 品質はエージェントによるテスト・静的検証・動的検証で担保する
- 人間はコードを見ない

> **思想**: CORE.md で「なぜ作るか」、ARCHITECTURE.md で「どう作るか」を書き、動くものから育てる。

## 開発フロー全体像

```
【起点】
AIチャットで壁打ち
または
/research → /debateで構想を練る

【設計】
1. /kickoff       → 構想書 → docs/CORE.md + docs/ARCHITECTURE.md → プロジェクト初期化

【デザイン → 実装】
2. /local         → ローカルサーバー起動
3. /mockup        → 代表1画面でデザイン決め → 確定後そのまま実装
4. /verify        → 品質チェック（静的検証・動的検証・テストを選択して実行）

【初回リリース】（コードがある状態で）
5. /infra-setup   → Git・GitHub・プラットフォーム・エラー監視・DBバックアップ等（後回し分は再実行可）
6. /release-setup → 環境変数・外部サービス・本番DB・スモークテスト
7. /audit         → リリース前の一括監査

【デプロイ】
8. /deploy

【育てる】（機能追加のサイクル）
実装 → /verify → /release-setup → /deploy safe → 繰り返し

【随時】
/audit
```

- 開発補助: `/backup`（git保存）、`/rollback`（巻き戻し）、`/gen-arch`（ドキュメント生成）、`/asset-preview`（アセット一覧HTML生成）、`/get-web-sounds`（BGM・SEのWEB調達）、`/gen-ai-pixels`（ローカルAI画像生成）
- 並列実行: `/parallel`（並列実装）、`/upgrade-team`（プロダクト改善：5視点調査＋実装）
- 調査・議論: `/research`（並列WebSearch調査）、`/debate`（複数のAIエージェントが議論）

## ドキュメント体系

```
docs/BRAINSTORM.md ─── 構想書（AIチャットや/debateで壁打ちして作成）
        ↓
docs/CORE.md ───────── Why（誰の課題を・どう解決するか）
        ↓
docs/ARCHITECTURE.md ─ How（技術スタック / 構成 / データモデル / API設計）
        ↓
CLAUDE.md + 実装コード
```

```
docs/current/ ─── コードの実態（/gen-arch で逆引き生成）
├── architecture.md
├── screen-flow.md
└── component-rules.md
```

## ツール一覧

### 毎回使う

| コマンド      | 概要                                    |
| --------- | ------------------------------------- |
| `/local`  | ローカルサーバーを起動してブラウザで開く                  |
| `/verify` | 品質チェック（静的検証・動的検証・テストを選択して並列実行）        |
| `/deploy` | git add → commit → push。`safe` で安全モード |

### 設計・初期構築

| コマンド | 概要 |
|---------|------|
| `/kickoff` | 対話で CORE.md + ARCHITECTURE.md を作成・編集 + プロジェクト初期化 |
| `/mockup` | 代表1画面でデザイン決め → 確定後そのまま実装（初回のみ） |
| `/infra-setup` | 本番の**箱を作る**（Git・GitHub・プラットフォーム・デプロイ設定・ドメイン・DBバックアップ・ブランチ・エラー監視）— 後回し分は再実行可 |
| `/release-setup` | 本番の**中身を整える**（環境変数・外部サービス・DB・スモークテスト）— 毎リリース |

### 並列実行・監査

| コマンド | 概要 |
|---------|------|
| `/parallel` | 並列実装。タスク分割してサブエージェントが同時に実装する |
| `/upgrade-team` | プロダクト改善。5視点で調査し改善提案（実装モード付き） |
| `/research` | 任意のテーマを多角的に並列WebSearch調査し、統合レポートを `.reports/` に出力 |
| `/debate` | 発散→収束の2フェーズ議論。前半フラット投げ合い、後半Builder×Expander×Criticで収束。`.reports/` に出力。researchレポートを参考資料として取り込み可。ユーザーは収束移行の拒否権＋発散終了の権限あり |
| `/audit` | 監査（コード・セキュリティ・設計・法令）を選択して並列実行 |

### アセット調達

| コマンド | 概要 |
|---------|------|
| `/get-web-sounds` | フリーBGM・SEの調査→自動選定→取得→配置→コード適用→クレジット管理 |
| `/gen-ai-pixels` | ローカルFLUX.1で画像アセットをテイスト確認→一括生成→配置→コード適用→クレジット管理 |

| コマンド        | 概要                                  |
| ----------- | ----------------------------------- |
| `/backup`   | 現在の状態をgitコミットで保存（pushしない）           |

| `/gen-arch` | コード実態からアーキテクチャドキュメントを自動生成           |
| `/asset-preview` | プロジェクトのアセット（画像・音声）をスキャンしてプレビューHTML生成 |

| コマンド | 概要 |
|---------|------|
| `/feature-trainer` | フロントエンドだけで小さなプロダクトを1日で作り切る（入門） |

## ツール詳細

### /mockup

**目的**: デザインの方向性を1画面で確定させ、後戻りを防ぐ

**フロー**:

```
1. docs/BRAINSTORM.md + CORE.md + ARCHITECTURE.md を読む
2. 代表1画面をHTML+CSSでモック作成（mockup/ディレクトリ）
3. ブラウザで確認 → フィードバック → 修正（ループ）
4. デザイン確定 → そのまま全画面を実装
```

**ポイント**:
- モックは1画面だけ（全画面作るのは実装）
- デザイントークン（色・フォント・余白）をCSS変数で定義 → 実装時に全画面に展開
- ロジック・JSはモックに入れない。見た目だけ
- 確定後、モックのデザインを崩さず実装する

---

### /kickoff

**成果物**: docs/CORE.md（Why）+ docs/ARCHITECTURE.md（How）+ プロジェクト初期化

- なければ新規作成（ヒアリング → 壁打ち → 出力）
- あれば編集モード（変更箇所のみ更新）

**セキュリティレベル**（ヒアリングで確定 → CLAUDE.md に記録）:

| レベル | 内容 |
|-------|------|
| `basic` | 認証なし、外部連携なし |
| `standard` | 認証あり、ユーザーデータ保存あり |
| `strict` | 決済・機密個人情報を扱う |

### /verify

実装後の品質チェックを一括実行。検証項目を選択して並列実行する。

**フロー**:

```
1. 検証項目を選択（複数選択可: テスト+静的検証（推奨） / テスト / 静的検証 / 動的検証）
2. テスト選択時 → テスト対象スコープを選択（直近の変更 / ファイル指定 / プロジェクト全体）
3. CORE.md / ARCHITECTURE.md からプロジェクト文脈を収集 → サブエージェントに渡す
4. 選択された検証を並列実行
5. 結果を統合して報告
```

| 検証項目 | エージェント | 対象 |
|---------|------------|------|
| 静的検証 | code-checker | 概念依存チェック・依存関係・リント・型チェック・ビルド確認（変更ファイル対象） |
| 動的検証 | visual-checker | ブラウザで画面確認 |
| テスト | test-checker | テスト生成＋実行（ユニット・インテグレーション・E2E）。3モード: 差分（デフォルト）/ スコープ指定 / 全体スキャン |

> **code-checker vs audit-code-checker**:
> code-checker（/verify 内）は変更ファイルの軽量チェック（実装直後用）。
> audit-code-checker（/audit 内）はプロジェクト全体の包括的監査（リリース前用）。

> **test-checker の3モード**:
> 差分モード（デフォルト）は `git diff` の変更箇所のみ。スコープ指定モードは指定ファイル/ディレクトリ対象。全体モードはプロジェクト全体をスキャンしてテストスイートを一括生成。

**使い分け**:

| 変更スコープ | 推奨 |
|------------|------|
| 小さな修正（typo、CSS微調整） | `/verify`（静的検証のみ） |
| 機能追加・ロジック変更 | `/verify`（静的検証） |
| UI変更を含む | `/verify`（静的検証 + 動的検証） |
| テスト整備 | `/verify`（テスト） |
| リリース前の包括的チェック | `/audit` |

### /audit

選択肢（複数選択可、デフォルト: 全て実行）:

| エージェント                     | 対象             |
| -------------------------- | -------------- |
| audit-code-checker         | コード品質・構造の包括的監査 |
| audit-security-checker     | 脆弱性・認証・秘密情報の検査 |
| audit-architecture-checker | 設計判断の妥当性レビュー   |
| audit-law-checker          | 法令遵守チェック       |

**使い分け**:

| 変更スコープ | 推奨 |
|------------|------|
| UI文言のみ変更 | コード監査のみ |
| 新機能追加 | コード + セキュリティ |
| 認証・決済まわりの変更 | コード + セキュリティ + 設計 |
| LP・広告表現の変更 | 法令 |
| リリース前 | 全て実行 |

### /parallel

**目的**: 実装指示を並列実行する。

**原則: 全操作worktree分離**

全てのサブエージェントは `isolation: "worktree"` で起動する。**例外なし。**
- タスクが1個でもworktreeで実行する
- mainは常にクリーンな状態を保つ（誰も直接触らない）
- 完了したものから順にmainへマージ

**フロー**:

```
1. 指示をタスクに分割（1〜5個、同一ファイル編集は1エージェントにまとめる）
2. 全エージェントを isolation: "worktree" で起動（1つのメッセージで並列起動）
   - 全タスクが独立 → Task並列起動（TeamCreate不要）
   - タスク間に依存関係あり → TeamCreate で管理
3. 完了したworktreeから順にmainへマージ
4. 衝突があれば解決し、結果を統合してユーザーに報告
```

### /get-web-sounds

**目的**: フリーBGM・SEの調査→自動選定→取得→配置→コード適用→クレジット管理を自動化

**フロー**:

```
0. プロジェクト分析 → 不足BGM・SEを特定 → 推奨提案
1. 🙋 何を取得する？（推奨から選択）
2. 一括調査・自動選定・DL・配置（場面ごとの確認なし）
3. コード適用（既存パターンに従って組み込み）
4. 🙋 動作確認する？
5. CREDITS.md 更新 → 完了報告
```

### /gen-ai-pixels

**目的**: ローカルFLUX.1で画像アセットをテイスト確認→一括生成→配置→コード適用→クレジット管理

**フロー**:

```
0. サーバー起動 + プロジェクト分析（並列）
1. プロンプト設計（MODEL_CHARACTERISTICS.md 参照）
2. テイスト確認
   既存アセットあり → フォルダを開いて 🙋「統一 or 変える」
     統一 → ステップ3へ直行
     変える → 5パターン生成 → 🙋 選択
   既存アセットなし → 5パターン生成 → 🙋 選択
3. 選んだテイストで残り一括生成
4. 配置（コピー・リネーム・リサイズ）
5. コード適用
6. CREDITS.md 更新 → 完了報告
7. /asset-preview でプレビュー更新
```

### /research → /debate

`/research` でレポート作成後、`/debate` で議論に持ち込める連携がある。

```
/research → .reports/*_research.md 作成 → /debate が自動検索して参考資料に取り込み
```

### /debate

**目的**: 任意のテーマを発散→収束の2フェーズで深掘り議論する

**フロー**:

```
Phase 1: 発散（役割なし・フラット）
│  2〜3行の短文で自由に投げ合い
│
├─ 誰かが「🔄 収束提案」+ 役割割り振り案を投稿
│   ├─ 全員合意（ユーザー参加中はユーザーの賛成も必須）→ Phase 2へ
│   ├─ 1人でも反対 → 継続（最低2往復後に再提案可）
│   └─ ユーザーが「終了」→ 発散の内容で結論ドキュメント作成して終了
│
▼
Phase 2: 収束（Builder → Expander → Critic）
│  Round 1: 素材整理 → 新角度投入 → 穴突き
│  Round 2〜: 採否回答+修正 → さらに新角度 → さらに深い問題
│
├─ 終了判定（Builderが判断、最低3ラウンド後）
│   全て解消 → 終了 / 未解決あり → 続行
│
▼
結論ドキュメント（Builderが作成、ファイル末尾）
  必須: 概要 / 決定事項 / リスクと対策
```

### トレーナースキル（学習系）

3段階の学習ステップ: **型を教わる → 型がスケールすると実感 → 自分で回す**

```
/feature-trainer ──→ /fullstack-trainer ──→ /solo-trainer ──→ 本番開発
   入門・1日            中級・2日             実践
```

#### /feature-trainer

フロントエンドのみの小さなプロダクトを1日で作り切り、開発フローの型を身体で覚える。
フロー: テーマ選択 → 環境構築 → スコープ固定 → /kickoff → 実装 → /verify → /deploy → 振り返り

#### /fullstack-trainer

DB + 外部API + Workersを加えた小さなフルスタックプロダクトを2日で作り切る。
feature-trainer と同じ型に DB設計フェーズを追加。テーマを変えて繰り返し使える。

#### /solo-trainer

自分でスキルを使って開発を回す練習。トレーナーは横にいて次の1手だけ指示する。

- **育てるモード**: CORE+ARCH+コードあり → 次の機能を提案 → 実装サイクルを繰り返す
- **作るモード**: 新規 or 設計途中 → テーマ決定 → /kickoff から開始

## リリース・障害対応

### 初回リリース

```
/infra-setup    → 箱を作る（後回し分は再実行可）
/release-setup  → 中身を整える
/audit          → 一括監査
/deploy         → 初回デプロイ
```

### 運用リリース

```
/release-setup  → 差分チェック＆設定補助
/audit          → 一括監査
/deploy safe    → 安全デプロイ
```

### 定期メンテナンス

`/audit` でコード監査・セキュリティ監査を選択して確認

### 常時監視（自動検知）

スキル・エージェントは手動実行。以下は設定すれば**自動で検知**してくれるサービス。

| サービス            | 何を検知                            | 設定場所                     | 通知先        | 無料枠         |
| --------------- | ------------------------------- | ------------------------ | ---------- | ----------- |
| **Sentry**      | 本番のJSランタイムエラー（スタックトレース・影響ユーザー数） | sentry.io                | Pushover   | 5,000イベント/月 |
| **UptimeRobot** | サイトダウン（HTTP応答なし）                | uptimerobot.com          | Pushover   | 50モニター・5分間隔 |
| **dependabot**  | 依存パッケージの脆弱性・更新                  | `.github/dependabot.yml` | メール        | 無制限         |

> Sentry はアプリ内で動くので「サイト自体が落ちた」は検知できない。UptimeRobot は外部から叩くので「JSバグ」は検知できない。補完関係。
> 緊急性の高いもの（Sentry・UptimeRobot）→ Pushover でプッシュ通知。急ぎでないもの（dependabot）→ メール。

### 障害対応

**検知**: Sentry（JSエラー）、UptimeRobot（サイトダウン）、ユーザー報告、`/deploy safe` のログチェック

```
1. 原因特定（ログ・エラー内容を確認）
2. 修正可能か？
   ├─ Yes → 修正 → /verify → /deploy safe
   └─ No  → /rollback
3. rules/learned.md に記録
```

### B. 検知カバレッジ

各検証手段が何を検知するかのマトリクス。

#### コード構造・ビルド

| 破壊の種類 | /verify 静的 | /verify 動的 | /verify テスト | audit コード |
|-----------|:---:|:---:|:---:|:---:|
| import/requireの破壊 | ✓ | - | - | ✓ |
| 関数シグネチャ変更 | - | - | ✓ | - |
| 型エラー / ビルドエラー | ✓ | - | - | - |
| リント違反 | ✓ | - | - | ✓ |
| 未使用ファイル・デッドコード | - | - | - | ✓ |
| 循環参照 | - | - | - | ✓ |
| Enum/定数の参照不整合 | - | - | - | ✓ |

#### UI・レイアウト

| 破壊の種類 | /verify 静的 | /verify 動的 | /verify テスト | audit コード |
|-----------|:---:|:---:|:---:|:---:|
| キャッシュバスティング未設定 | ✓ | - | - | ✓ |
| CSSセレクタ不一致 | ✓ | ✓ | - | - |
| レイアウト崩れ（ピクセル） | - | ✓ | - | - |
| レイアウト崩れ（要素・遷移） | - | ✓ | ✓ | - |
| コンソールエラー | - | ✓ | ✓ | - |
| JSランタイムエラー | △ | ✓ | ✓ | - |
